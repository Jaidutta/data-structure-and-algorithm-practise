1️⃣ Introduction & Core Concepts

What is Two Pointers?

The Two Pointers technique uses two indices (or "pointers") to traverse a data structure simultaneously, optimizing solutions that would otherwise require nested loops.

👤 The Human Analogy

Imagine you have a row of 8 lockers, each with a number on it. Your task: find two lockers that add up to 10.

❌ Without Two Pointers — The Naive Way

You send one person (🧍 Alex). Alex picks locker 1, then walks to every other locker to check if it pairs with locker 1. Then goes back, picks locker 2, walks to every other locker again. And so on...

Why is this smart?
Because the array is sorted, Alex and Blake can reason at every step:
• Sum too big → Blake moves left (reduce the right value)
• Sum too small → Alex moves right (increase the left value)
• Sum exact → done! 🎉

They never need to re-check anything. Each step eliminates an entire column of possibilities. With 1000 lockers: only up to 1000 checks instead of 1,000,000. That's O(N) vs O(N²).

The Decision Flowchart

Check if your problem has these characteristics:

💡 Original order does not matter
↓ YES
Does it involve pairs/triplets OR
rearranging/merging in-place?
Examples:
• pair(i, j), triplet(i, j, k), quadruple(i, j, k, l)
• Rearrange: 0/1/2 or red/blue/green balls in-order without using any extra space
• Detect cycle in a linkedlist
↓ YES
Is there a space constraint
(O(1) space needed)?
↓ YES
✅ USE TWO POINTERS!

✅ When to Use

  • ✓ Data is sorted or can be sorted
  • ✓ Need to find pairs, triplets, or quadruplets
  • ✓ Problem involves rearranging elements
  • ✓ Need to remove duplicates
  • ✓ Merge operations in-place
  • ✓ Constraint: "Do not use extra space"
  • ✓ Need to optimize from O(N²) to O(N)

❌ When NOT to Use

  • ✗ Unsorted array and must return indices → Use HashMap
  • ✗ Need ALL subsets → Use Backtracking
  • ✗ Optimization problems (min/max) → Use DP
  • ✗ Graph traversal → Use DFS/BFS
  • ✗ Only need to count something → May use DP or Math

3️⃣ Two Sum (Sorted Array) - Foundation

Problem: Find two numbers in a sorted array that add up to target
Example: arr = [1, 3, 5, 7, 9], target = 12 → Output: [3, 9]

Why Two Pointers Works

Array is sorted: smallest at start, largest at end
• If sum too small → need bigger number (move left forward)
• If sum too large → need smaller number (move right backward)

🧠 Core Logic

// Two Sum Template
L = 0,  R = n -1

while (L < R) {
    sum = arr[L] + arr[R]
    
    if (sum == target)
        return [arr[L], arr[R]]
    
    else if (sum < target)
        L++   // Need larger number
    
    else
        R--   // Need smaller number
}

/**
 * Two Sum - Sorted Array
 * 
 * Problem: Find two numbers that add up to target in sorted array
 * 
 * Approach: Two Pointers from opposite ends
 * - left pointer starts at smallest element (index 0)
 * - right pointer starts at largest element (index n-1)
 * - Move pointers based on sum comparison with target
 * 
 * Why it works:
 * - Array is SORTED, so we know smallest and largest values
 * - If sum too small → need bigger number → move left forward
 * - If sum too large → need smaller number → move right backward
 * - Pointers eventually meet or find the answer
 * 
 * Time: O(N) - single pass through array
 * Space: O(1) - only two pointers used
 */
public int[] twoSumSorted(int[] arr, int target) {
    // Initialize left pointer at start (smallest element)
    int left = 0;
    
    // Initialize right pointer at end (largest element)
    int right = arr.length - 1;
    
    // Keep searching while pointers haven't crossed
    // Stop when left >= right (no more pairs to check)
    while (left < right) {
        // Calculate sum of current pair
        int sum = arr[left] + arr[right];
        
        // Case 1: Found the target sum!
        if (sum == target) {
            // Return the indices (or values depending on problem)
            return new int[]{left, right};
        } 
        
        // Case 2: Sum is too small
        // We need a LARGER sum, so move left pointer forward
        // This gives us a bigger number (array is sorted!)
        else if (sum < target) {
            left++;
        } 
        
        // Case 3: Sum is too large
        // We need a SMALLER sum, so move right pointer backward
        // This gives us a smaller number (array is sorted!)
        else {
            right--;
        }
    }
    
    // If we exit the loop, no pair was found
    // Return [-1, -1] to indicate no solution exists
    return new int[]{-1, -1};
}
Time: O(N) Space: O(1)

💻 Java Code (Fully Commented)

/**
 * Merge Two Sorted Arrays
 * Time: O(N + M), Space: O(N + M)
 */
public int[] mergeSortedArrays(int[] A, int[] B) {
    int[] result = new int[A.length + B.length];
    int i = 0, j = 0, k = 0;
    
    // Merge while both have elements
    while (i < A.length && j < B.length) {
        if (A[i] <= B[j]) {
            result[k++] = A[i++];
        } else {
            result[k++] = B[j++];
        }
    }
    
    // Copy remaining from A
    while (i < A.length) result[k++] = A[i++];
    
    // Copy remaining from B
    while (j < B.length) result[k++] = B[j++];
    
    return result;
}

4️⃣ Squares of Sorted Array

Problem: Given sorted array with negatives, return sorted squares
Example: [-4, -1, 0, 3, 10] → [0, 1, 9, 16, 100]

Why Simple Squaring Breaks Sorting

Original:  [-4, -1,  0,  3, 10]
After ^2:  [16,  1,  0,  9, 100]  ❌ NOT SORTED!

Why? -4 squared = 16 (larger than 3 squared = 9)

Squaring -ve elements gives us positive numbers and as a result it becomes tricky since they are sorted in descending order

Solution: Split and Merge

Key Insight

The array has TWO sorted parts:

  1. Negatives: Read right-to-left and square → ascending (no reverse needed)
  2. Positives: Already ascending → square on the fly during merge (no extra list needed)

📊 Visual Example

Input: [-4, -1, 0, 3, 10]

Step 1: Split
neg = [-4, -1]      pos = [0, 3, 10]

Step 2: Set pointers
i = 1 (END of neg)  j = 0 (START of pos)
↑ reading neg right-to-left gives ascending squares — no reverse() needed!
↑ pos is already ascending — no posSquared list needed, square on the fly!

Step 3: Merge (squaring on the fly)
i=1: (-1)²=1  vs  j=0: 0²=0   → 0 smaller  → result[0]=0,  j=1
i=1: (-1)²=1  vs  j=1: 3²=9   → 1 smaller  → result[1]=1,  i=0
i=0: (-4)²=16 vs  j=1: 3²=9   → 9 smaller  → result[2]=9,  j=2
i=0: (-4)²=16 vs  j=2: 10²=100 → 16 smaller → result[3]=16, i=-1
neg exhausted → drain pos: result[4]=100

Final: [0, 1, 9, 16, 100] ✓
Time: O(N) Space: O(N)

💻 Java Code

💡 Key Observations:
  • Positives squared → stay ascending → no extra list needed, square on the fly
  • Negatives read right-to-left and squared → ascending → no reverse() needed
// Approach 1: Split + Merge
// Time: O(N)  |  Space: O(N)

public int[] sortedSquares(int[] nums) {
    int[] neg = new int[nums.length];
    int[] pos = new int[nums.length];
    int negSize = 0, posSize = 0;

    for (int num : nums) {
        if (num < 0) neg[negSize++] = num;
        else         pos[posSize++] = num;
    }

    // i starts at END of neg → right-to-left gives ascending squares (no reverse needed)
    // j starts at START of pos → already ascending, square on the fly
    int[] result = new int[nums.length];
    int i = negSize - 1;
    int j = 0;
    int k = 0;

    while (i >= 0 && j < posSize) {
        int negVal = neg[i] * neg[i];
        int posVal = pos[j] * pos[j];

        if (negVal <= posVal) { result[k++] = negVal; i--; }
        else                  { result[k++] = posVal; j++; }
    }

    while (i >= 0)      { result[k++] = neg[i] * neg[i]; i--; }
    while (j < posSize) { result[k++] = pos[j] * pos[j]; j++; }

    return result;
}

 * i=1: (-1)²=1  vs 3²=9   → result[1]=1,  i=0
 * i=0: (-4)²=16 vs 3²=9   → result[2]=9,  j=2
 * i=0: (-4)²=16 vs 10²=100 → result[3]=16, i=-1

💻 Optimal: Two Pointers on Original Array

✅ No helper lists at all. The largest square always lives at one of the two ends — compare both ends, place the larger one from the right. 
// Approach 2: Two Pointers on original array  ← Most Optimal
// Time: O(N)  |  Space: O(1) extra
public int[] sortedSquares(int[] nums) {
    int n = nums.length;
    int[] result = new int[n];

    int L = 0;
    int R = n - 1;
    int k = n - 1;  // fill result right to left

    while (L <= R) {
        int leftSq  = nums[L] * nums[L];
        int rightSq = nums[R] * nums[R];

        if (leftSq > rightSq) { result[k--] = leftSq;  L++; }
        else                  { result[k--] = rightSq; R--; }
    }

    return result;
}

💻 Java Code (Fully Commented)

/**
 * Remove Duplicates from Sorted Array
 * 
 * Problem: Remove duplicates in-place, return count of unique elements
 * Input: [1, 1, 2, 2, 3]
 * Output: 3 (array becomes [1, 2, 3, _, _])
 * 
 * Approach: Slow and Fast Pointers (Officer & CM analogy)
 * - Officer (slow): Tracks position of last unique element placed
 * - CM (fast): Scans ahead to find new unique elements
 * 
 * Key Insight:
 * - In sorted array, duplicates are GROUPED together
 * - Compare current with previous: if different, it's unique!
 * - Officer only moves when a new unique element is found
 * 
 * Time: O(N) - single pass through array
 * Space: O(1) - in-place modification
 */
public int removeDuplicates(int[] nums) {
    // Edge case: empty array has 0 unique elements
    if (nums.length == 0) return 0;
    
    // Officer pointer: tracks last unique element position
    // Starts at 0 because first element is ALWAYS unique
    int officer = 0;
    
    // CM pointer: scans array starting from second element
    // We use the loop variable 'cm' as our fast pointer
    for (int cm = 1; cm < nums.length; cm++) {
        
        // Check if current element is different from previous
        // If nums[cm] != nums[cm-1], we found a UNIQUE element!
        if (nums[cm] != nums[cm - 1]) {
            
            // Move officer forward to next empty slot
            officer++;
            
            // Place the unique element at officer's position
            // This builds the unique array at the front
            nums[officer] = nums[cm];
        }
        
        // If nums[cm] == nums[cm-1], it's a DUPLICATE
        // CM just continues scanning (loop auto-increments cm)
        // Officer stays put - doesn't waste a slot on duplicates
    }
    
    // Return count of unique elements
    // Officer is at last unique position (0-indexed)
    // So count = officer + 1
    // Example: if officer = 2, we have elements at [0,1,2] = 3 elements
    return officer + 1;
}

/*
 * Example Trace: [1, 1, 2, 2, 3]
 * 
 * Initial: officer=0, cm=1
 *          [1, 1, 2, 2, 3]
 *           ^  ^
 *           O  C
 *          nums[1] == nums[0]? → YES (duplicate), continue
 * 
 * Step 1: officer=0, cm=2
 *         [1, 1, 2, 2, 3]
 *          ^     ^
 *          O     C
 *         nums[2] != nums[1]? → YES (unique!)
 *         officer++ → officer=1
 *         nums[1] = nums[2] = 2
 *         Array: [1, 2, 2, 2, 3]
 * 
 * Step 2: officer=1, cm=3
 *         [1, 2, 2, 2, 3]
 *             ^     ^
 *             O     C
 *         nums[3] == nums[2]? → YES (duplicate), continue
 * 
 * Step 3: officer=1, cm=4
 *         [1, 2, 2, 2, 3]
 *             ^        ^
 *             O        C
 *         nums[4] != nums[3]? → YES (unique!)
 *         officer++ → officer=2
 *         nums[2] = nums[4] = 3
 *         Array: [1, 2, 3, 2, 3]
 * 
 * Result: officer = 2, return 2+1 = 3
 *         First 3 elements are unique: [1, 2, 3]
 */
Time: O(N) Space: O(1)

6️⃣ Dutch National Flag (Sort Colors)

Problem: Sort array of 0s, 1s, 2s in ONE PASS
Example: [2, 0, 2, 1, 1, 0] → [0, 0, 1, 1, 2, 2]
Constraint: O(N) time, O(1) space, no library sort()

🧠 How the Algorithm Works

The idea is to maintain three pointers that divide the array into four regions at all times: 

[  0s  |  1s  |  unknown  |  2s  ]
   ↑      ↑        ↑          ↑
   0    low       mid        high   n-1
  1. 0 … low-1 → confirmed 0s (left wall, grows rightward)
  2. low … mid-1 → confirmed 1s (middle, grows rightward)
  3. mid … high → unknown — this is what mid scans through
  4. high+1 … n-1 → confirmed 2s (right wall, grows leftward)

The goal: keep shrinking the unknown region until it disappears (mid > high).
mid is the scanner — it looks at arr[mid] and decides what to do:

0️⃣

arr[mid] == 0

0 belongs at the left.
Swap with low.
low++ (0s zone grew)
mid++ (safe — low came from processed region, so swapped value is 0 or 1, not unknown)

1️⃣

arr[mid] == 1

1 belongs in the middle.
Already in the right place — no swap needed.
Just mid++ to expand the 1s zone and move on.

2️⃣

arr[mid] == 2

2 belongs at the right.
Swap with high.
high-- (2s zone grew)
mid stays! — high came from the unknown region, so we must check the swapped value before moving on.

🔑 The One Tricky Rule

When swapping with low → mid++ is safe because low is always in the already-processed region (contains 0 or 1 — both fine at mid's position).

When swapping with high → mid must stay because high came from the unknown region. It could be 0, 1, or 2 — we don't know yet and must check it next.

arr[mid] Action Move mid? Why?

 * Problem: Sort array of 0s, 1s, 2s in ONE PASS
 * Input: [2, 0, 2, 1, 1, 0]
 * Output: [0, 0, 1, 1, 2, 2]
 * 
 * Constraint: O(N) time, O(1) space, no library sort()
 * 
 * Approach: Three Pointers to Partition Array into 4 Regions
 * 
 * Array Layout:
 * [  0s  |  1s  | unknown |  2s  ]
 *    ↑      ↑       ↑        ↑
 *    0    low    mid      high   n-1
 * 
 * Regions:
 * 1. [0 to low-1]: Contains all 0s (SORTED)
 * 2. [low to mid-1]: Contains all 1s (SORTED)
 * 3. [mid to high]: UNKNOWN (needs processing)
 * 4. [high+1 to end]: Contains all 2s (SORTED)
 * 
 * Goal: Shrink the unknown region to zero
 * 
 * Rules:
 * - If arr[mid] == 0: Belongs at start, swap with low
 * - If arr[mid] == 1: Already in correct place, just move mid
 * - If arr[mid] == 2: Belongs at end, swap with high
 * 
 * Time: O(N) - single pass
 * Space: O(1) - in-place
 */
public void sortColors(int[] nums) {
    // low: boundary of 0s region (end of 0s)
    // Everything from 0 to low-1 should be 0s
    int low = 0;
    
    // mid: current scanner/processing pointer
    // This scans the unknown region
    int mid = 0;
    
    // high: boundary of 2s region (start of 2s)
    // Everything from high+1 to end should be 2s
    int high = nums.length - 1;
    
    // Process until mid crosses high
    // Note: mid <= high (NOT mid < high)
    // We need to process the element at high too!
    while (mid <= high) {
        
        // ===== CASE 1: Current element is 0 =====
        // 0 belongs at the START (with other 0s)
        if (nums[mid] == 0) {
            // Swap arr[mid] with arr[low]
            // This moves the 0 to the 0s region
            int temp = nums[low];
            nums[low] = nums[mid];
            nums[mid] = temp;
            
            // Increment low: 0s region grew by 1
            low++;
            
            // Increment mid: We can safely move forward
            // WHY? Because arr[low] was either 0 or 1 (from processed region)
            // After swap, arr[mid] is now 0 or 1, which is fine in 1s region
            mid++;
        } 
        
        // ===== CASE 2: Current element is 1 =====
        // 1 is already in the CORRECT position (middle region)
        else if (nums[mid] == 1) {
            // Just move mid forward, nothing to swap
            // The 1s region automatically grows
            mid++;
        } 
        
        // ===== CASE 3: Current element is 2 =====
        // 2 belongs at the END (with other 2s)
        else {  // nums[mid] == 2
            // Swap arr[mid] with arr[high]
            // This moves the 2 to the 2s region
            int temp = nums[high];
            nums[high] = nums[mid];
            nums[mid] = temp;
            
            // Decrement high: 2s region grew by 1
            high--;
            
            // CRITICAL: Do NOT increment mid!
            // WHY? Because arr[high] came from UNKNOWN region
            // It could be 0, 1, or 2 - we don't know yet!
            // We need to CHECK it in the next iteration
            // This is the KEY difference from Case 1
        }
    }
    
    // When loop ends, mid > high
    // Unknown region is now empty - array is sorted!
}

/*
 * Example Trace: [2, 0, 2, 1, 1, 0]
 * 
 * Initial: low=0, mid=0, high=5
 *          [2, 0, 2, 1, 1, 0]
 *           ^              ^
 *          L/M             H
 * 
 * Step 1: arr[mid]=2, swap with high
 *         Swap arr[0] and arr[5]: 2 ↔ 0
 *         [0, 0, 2, 1, 1, 2]
 *          ^           ^
 *         L/M          H
 *         high-- → high=4
 *         mid stays at 0 (need to check the 0 we just got)
 *

 *         [0, 0, 2, 1, 1, 2]

 *         low++ → low=1

 *         Swap arr[1] and arr[1]: 0 ↔ 0 (no change)

 *             ^  ^     ^
 *             L  M     H

 *         Swap arr[2] and arr[4]: 2 ↔ 1

🔑 Critical Insight

Why don't we increment mid after swapping with high?

When we swap arr[mid] with arr[high]:
• We DON'T KNOW what arr[high] was (it's from unknown region)
• Could be 0, 1, or 2 → We need to CHECK it!

When we swap arr[mid] with arr[low]:
• We KNOW arr[low] was either 0 or 1 (processed region)
• Safe to move both pointers forward

Time: O(N) Space: O(1)

6.5️⃣ Two Sum - Handling Duplicates

Problem: Find all unique pairs in a sorted array that sum to target
Example: arr = [1, 1, 2, 2, 3, 4, 5], target = 6
Challenge: Array has duplicates — avoid counting the same pair twice!

🔄 EVOLUTION: 2Sum → Find ALL Unique Pairs

Change: Find ALL unique pairs instead of just one
Example: arr = [-2, -1, 0, 0, 1, 2], target = 0
Output: [[-2,2], [-1,1], [0,0]]

💡 What to Add

Same base logic, ADD duplicate skipping:

When you find a match:

  1. Store the pair
  2. Skip duplicate lefts: while (left < right && arr[left] == arr[left+1]) left++;
  3. Skip duplicate rights: while (left < right && arr[right] == arr[right-1]) right--;
  4. Move both: left++; right--;

🎯 The Key Challenge

When there are duplicates in the array, naive two pointers would return:
[1, 5], [1, 5], [2, 4], [2, 4] ❌

But we want unique pairs only:
[1, 5], [2, 4] ✓

💡 Solution: Skip Duplicates After Finding a Pair

After finding a valid pair, we need to skip all duplicate values at both L and R pointers:

while (L < R && arr[L] == arr[L+1]) L++;
while (L < R && arr[R] == arr[R-1]) R--;

🧠 Core Logic

// Two Sum with Duplicates Template
L = 0,  R = n -1
result = []

while (L < R) {
    sum = arr[L] + arr[R]
    
    if (sum == target) {
        result.add([arr[L], arr[R]])
        
        // SKIP DUPLICATES at both ends
        L++
        R--
        while (L < R && arr[L] == arr[L-1]) L++
        while (L < R && arr[R] == arr[R+1]) R--
    }
    else if (sum < target)
        L++
    else
        R--
}

return result

💻 Java Code (Fully Commented)

/**
 * Two Sum - Find All Unique Pairs (With Duplicates)
 * 
 * Problem: Find all unique pairs that sum to target
 * Input: sorted array with duplicates
 * Output: List of unique pairs
 * 
 * Key Challenge: Skip duplicates to avoid counting same pair twice
 * 
 * Time: O(N) - single pass with two pointers
 * Space: O(1) - only store results
 */
public int[][] twoSum(int[] arr, int target) {
    List<List<Integer>> result = new ArrayList<>();
    int L = 0, R = arr.length - 1;

    while (L < R) {
        int sum = arr[L] + arr[R];

        if (sum == target) {
            result.add(Arrays.asList(arr[L], arr[R]));
            L++;
            R--;

            // CRITICAL: Skip all duplicates at L
            while (L < R && arr[L] == arr[L - 1]) L++;

            // CRITICAL: Skip all duplicates at R
            while (L < R && arr[R] == arr[R + 1]) R--;

        } else if (sum < target) {
            L++;
        } else {
            R--;
        }
    }

    // Convert to int[][]
    int[][] resultArray = new int[result.size()][2];
    for (int i = 0; i < result.size(); i++) {
        resultArray[i][0] = result.get(i).get(0);
        resultArray[i][1] = result.get(i).get(1);
    }
    return resultArray;
}

/*
 * Example Trace: arr = [1, 1, 2, 2, 3, 4, 5], target = 6
 * 
 * Initial: L=0, R=6
 * Step 1: arr[0] + arr[6] = 1 + 5 = 6 ✓
 *         Add [1, 5] to result
 *         Skip duplicates: L moves from 0 → 1 → 2 (skip the second 1)
 *         R moves from 6 → 5 (no duplicate)
 *         Now: L=2, R=5
 * 
 * Step 2: arr[2] + arr[5] = 2 + 4 = 6 ✓
 *         Add [2, 4] to result
 *         Skip duplicates: L moves from 2 → 3 → 4 (skip the second 2)
 *         R moves from 5 → 4 (no duplicate)
 *         Now: L=4, R=4 → L >= R → Stop
 * 
 * Result: [[1, 5], [2, 4]]
 */

✅ Key Takeaways

  • Why skip duplicates? To ensure we return unique pairs only
  • When to skip? After finding a valid pair, before moving pointers
  • How to skip? Use while loops: while (arr[L] == arr[L+1]) L++;
  • Both sides? Yes! Must skip duplicates at both L and R

⚠️ Common Mistakes

  • ❌ Forgetting to skip duplicates → returns duplicate pairs
  • ❌ Only skipping at L or R (not both) → still returns duplicates
  • ❌ Moving L++ or R-- before skipping → off-by-one errors
  • ❌ Not checking L < R in skip loops → array out of bounds
Time: O(N) Space: O(1)

7️⃣ 3Sum Problems (Triplets)

Common Pattern for All Triplet Problems

1. Sort array O(N log N)
2. Fix one element with loop i
3. Use Two Pointers on remaining array

🎯 EVOLUTION: 3Sum = Fix One + 2Sum

💡 The Key Insight

3Sum is just 2Sum called multiple times!

Fix arr[i], then find two numbers summing to -arr[i] in the rest
The inner loop is IDENTICAL to 2Sum with duplicates

💡 The Strategy

  1. Outer loop: Fix element at index i
  2. Inner: Run 2Sum on arr[i+1...n-1] with target = -arr[i]
  3. Skip duplicates at 3 places:
    • ① Skip duplicate i: if (i > 0 && arr[i] == arr[i-1]) continue;
    • ② Skip duplicate left (same as 2Sum all pairs)
    • ③ Skip duplicate right (same as 2Sum all pairs)

Problem: 3Sum (Find Triplets Summing to Zero)

Input: [-1, 0, 1, 2, -1, -4]
Output: [[-1, -1, 2], [-1, 0, 1]]
Constraint: No duplicate triplets

🧠 Core Logic

// 3Sum Template (extends Two Sum with Duplicates)
sort(arr)
result = []

for i = 0 to n-3:
    // Skip duplicate i values
    if (i > 0 && arr[i] == arr[i-1]) continue
    
    // Fix arr[i], find Two Sum for remaining
    L = i + 1
    R = n -1
    target = -arr[i]  // Want arr[i] + arr[L] + arr[R] = 0
    
    while (L < R) {
        sum = arr[L] + arr[R]
        
        if (sum == target) {
            result.add([arr[i], arr[L], arr[R]])
            
            // SKIP DUPLICATES at L and R
            L++
            R--
            while (L < R && arr[L] == arr[L-1]) L++
            while (L < R && arr[R] == arr[R+1]) R--
        }
        else if (sum < target)
            L++
        else
            R--
    }

return result

💻 Java Code (Fully Commented)

/**
 * 3Sum - Find All Unique Triplets that Sum to Zero
 * 
 * Problem: Find all unique triplets [a, b, c] where a + b + c = 0
 * Input: [-1, 0, 1, 2, -1, -4]
 * Output: [[-1, -1, 2], [-1, 0, 1]]
 * 
 * Constraint: Solution set must NOT contain duplicate triplets
 * 
 * Approach: Fix one element, then use Two Pointers on rest
 * 1. Sort array first (enables two pointers + duplicate skipping)
 * 2. Loop through array with index i (this is the "fixed" element)
 * 3. For remaining array, use Two Pointers to find pairs that sum to -arr[i]
 * 4. Skip duplicates at THREE places: i, left, right
 * 
 * Why this works:
 * - Fixing arr[i] reduces 3Sum to 2Sum problem
 * - Sorted array allows efficient two pointers
 * - Sorting groups duplicates, making them easy to skip
 * 
 * Time: O(N²) - O(N log N) for sort + O(N) loop × O(N) two pointers
 * Space: O(1) - not counting output array
 */
public List<List<Integer>> threeSum(int[] nums) {
    // Step 1: Sort the array
    // This enables: two pointers approach + easy duplicate skipping
    Arrays.sort(nums);
    
    // Result list to store all unique triplets
    List<List<Integer>> result = new ArrayList<>();
    
    // Step 2: Loop through array, fixing one element at a time
    // Stop at n-2 because we need at least 2 more elements for left and right
    for (int i = 0; i < nums.length - 2; i++) {
        
        // ===== DUPLICATE SKIP 1: Skip duplicate 'i' values =====
        // If current element same as previous, we'd get duplicate triplets
        // Example: [-1, -1, 0, 1] - if we process both -1s, we get duplicates
        // Check i > 0 to avoid index out of bounds on first iteration
        if (i > 0 && nums[i] == nums[i - 1]) {
            continue;  // Skip this iteration
        }
        
        // Step 3: Set up two pointers for remaining array
        // left: starts right after i
        int left = i + 1;
        
        // right: starts at end of array
        int right = nums.length - 1;
        
        // We want: arr[i] + arr[left] + arr[right] = 0
        // Rearranging: arr[left] + arr[right] = -arr[i]
        // So target for two pointers is -arr[i]
        int target = -nums[i];
        
        // Step 4: Two pointers approach on remaining array
        while (left < right) {
            // Calculate sum of the two pointers
            int sum = nums[left] + nums[right];
            
            // ===== CASE 1: Found a valid triplet! =====
            if (sum == target) {
                // Add triplet to result
                // [nums[i], nums[left], nums[right]] sums to 0
                result.add(Arrays.asList(nums[i], nums[left], nums[right]));
                
                // ===== DUPLICATE SKIP 2: Skip duplicate 'left' values =====
                // Move left forward while next element is same
                // Example: [-1, 0, 0, 0, 1] - skip the extra 0s
                while (left < right && nums[left] == nums[left + 1]) {
                    left++;
                }
                
                // ===== DUPLICATE SKIP 3: Skip duplicate 'right' values =====
                // Move right backward while previous element is same
                while (left < right && nums[right] == nums[right - 1]) {
                    right--;
                }
                
                // Move both pointers after finding a triplet
                // (and after skipping duplicates)
                left++;
                right--;
            } 
            
            // ===== CASE 2: Sum too small =====
            // Need a larger sum, move left forward (array is sorted!)
            else if (sum < target) {
                left++;
            } 
            
            // ===== CASE 3: Sum too large =====
            // Need a smaller sum, move right backward
            else {
                right--;
            }
        }
    }
    
    // Return all unique triplets found
    return result;
}

/*
 * Example Trace: [-1, 0, 1, 2, -1, -4]
 * After sort: [-4, -1, -1, 0, 1, 2]
 * 
 * Iteration i=0: arr[i]=-4, target=4
 * [-4, -1, -1, 0, 1, 2]
 *   ^   L           R
 * sum = -1+2 = 1 < 4 → left++
 * sum = -1+2 = 1 < 4 → left++
 * sum = 0+2 = 2 < 4 → left++
 * sum = 1+2 = 3 < 4 → left++
 * left >= right, no solution for i=0
 * 
 * Iteration i=1: arr[i]=-1, target=1
 * [-4, -1, -1, 0, 1, 2]
 *       ^   L      R
 * sum = -1+2 = 1 == 1 ✓ Found triplet: [-1, -1, 2]
 * Skip duplicates, move pointers
 * 
 * Continue finding: [-1, 0, 1]
 * 
 * Iteration i=2: arr[i]=-1 == arr[1]=-1 → Skip duplicate i
 * 
 * Final result: [[-1, -1, 2], [-1, 0, 1]]
 * 
 * KEY INSIGHT: Three places to skip duplicates!
 * 1. Duplicate i (outer loop)
 * 2. Duplicate left (after finding triplet)
 * 3. Duplicate right (after finding triplet)
 * Without this, we'd get duplicate triplets in output
 */
Time: O(N²) Space: O(1)

💻 3Sum Closest - Java Code (Fully Commented)

🎯 Key Difference from Regular 3Sum

3Sum: Find triplets that sum to EXACTLY zero
3Sum Closest: Find triplet that sums CLOSEST to target

Strategy: Track the minimum distance from target throughout the search!

🧠 Core Logic

// 3Sum Closest Template (same loop structure as 3Sum)
sort(arr)
closestSum = arr[0] + arr[1] + arr[2]  // Initial guess

for i = 0 to n-3:
    // Fix arr[i], use Two Pointers on remaining
    L = i + 1
    R = n -1
    
    while (L < R) {
        currentSum = arr[i] + arr[L] + arr[R]
        
        // Update if this sum is closer to target
        if (|currentSum - target| < |closestSum - target|)
            closestSum = currentSum
        
        // Move pointers based on comparison (same as 3Sum)
        if (currentSum < target)
            L++   // Need larger sum
        else if (currentSum > target)
            R--   // Need smaller sum
        else
            return currentSum  // Exact match, can't get closer!
    }

return closestSum

🔑 Key Insight: No Duplicate Skipping Needed!

Unlike regular 3Sum, we DON'T need to skip duplicates because:
• We're finding just ONE triplet (the closest), not ALL triplets
• Duplicate combinations will have the same sum anyway
• No need to avoid returning duplicate results

/**
 * 3Sum Closest
 * 
 * Problem: Find triplet with sum CLOSEST to target
 * Input: nums = [-1, 2, 1, -4], target = 1
 * Output: 2 (triplet [-1, 1, 2] sums to 2, closest to 1)
 * 
 * Approach: Same structure as 3Sum but track closest sum
 * 1. Sort array (enables two pointers)
 * 2. Fix element i
 * 3. Use two pointers to find closest sum
 * 4. Track minimum distance from target throughout
 * 
 * Key Difference from 3Sum:
 * - 3Sum: Find exact match (sum == 0)
 * - 3Sum Closest: Find minimum distance (track |sum - target|)
 * 
 * Time: O(N²) - O(N log N) sort + O(N) outer * O(N) inner
 * Space: O(1) - only using pointers
 */
public int threeSumClosest(int[] nums, int target) {
    // Step 1: Sort the array
    // This enables two pointers approach
    Arrays.sort(nums);
    
    // Initialize closestSum with first possible triplet
    // This gives us a baseline to compare against
    int closestSum = nums[0] + nums[1] + nums[2];
    
    // Step 2: Fix first element with loop
    // Stop at length-2 because we need 2 more elements for left/right
    for (int i = 0; i < nums.length - 2; i++) {
        // Step 3: Set up two pointers for remaining array
        int left = i + 1;           // Start after fixed element
        int right = nums.length - 1; // Start at end
        
        // Step 4: Two pointers search
        while (left < right) {
            // Calculate current triplet sum
            int currentSum = nums[i] + nums[left] + nums[right];
            
            // ===== KEY: Update closestSum based on distance =====
            // Compare absolute distances from target
            // |currentSum - target| vs |closestSum - target|
            if (Math.abs(currentSum - target) < Math.abs(closestSum - target)) {
                closestSum = currentSum;
            }
            
            // ===== OPTIMIZATION: Early exit if exact match =====
            // If we find exact target, distance = 0 (can't get closer!)
            if (currentSum == target) {
                return currentSum; // Perfect match, return immediately
            }
            
            // ===== Move pointers based on comparison with TARGET =====
            // (NOT based on closestSum - still compare to target!)
            
            // Case 1: Current sum is less than target
            // Need larger sum → move left pointer forward
            if (currentSum < target) {
                left++;
            } 
            // Case 2: Current sum is greater than target
            // Need smaller sum → move right pointer backward
            else {
                right--;
            }
        }
    }
    
    // Return the closest sum we found
    return closestSum;
}

/* 
 * Complete Trace: nums = [-1, 2, 1, -4], target = 1
 * After sort: [-4, -1, 1, 2]
 * 
 * Initialize: closestSum = -4 + (-1) + 1 = -4 (distance = 5)
 * 
 * i=0 (fixed=-4):
 *   left=1, right=3: [-4, -1, 2] → sum=-3, distance=4 → UPDATE to -3
 *   left=2, right=3: [-4, 1, 2] → sum=-1, distance=2 → UPDATE to -1
 *   left > right → inner loop ends
 * 
 * i=1 (fixed=-1):
 *   left=2, right=3: [-1, 1, 2] → sum=2, distance=1 → UPDATE to 2 ✓
 *   currentSum (2) > target (1) → right--
 *   left > right → inner loop ends
 * 
 * i=2 would start, but i < 2 is false → outer loop ends
 * 
 * Return closestSum = 2
 * 
 * FINAL ANSWER: Triplet [-1, 1, 2] has sum 2, closest to target 1
 */

/**
 * Triplets with Smaller Sum
 * 
 * Problem: COUNT triplets where sum < target
 * Input: nums = [-1, 0, 2, 3], target = 3
 * Output: 2 (triplets: [-1, 0, 2] and [-1, 0, 3])
 * 
 * Approach: Same structure but with counting optimization
 * - If sum < target: ALL elements from left to right work!
 * - This is because array is sorted
 * - Add (right - left) to count immediately
 * 
 * KEY INSIGHT:
 * If arr[i] + arr[left] + arr[right] < target
 * Then arr[i] + arr[left] + ANY element between left and right < target
 * Because array is sorted, all middle elements ≤ arr[right]
 * 
 * Time: O(N²)
 * Space: O(1)
 */
public int threeSumSmaller(int[] nums, int target) {
    Arrays.sort(nums);
    int count = 0;
    
    for (int i = 0; i < nums.length - 2; i++) {
        int left = i + 1;
        int right = nums.length - 1;
        
        while (left < right) {
            int sum = nums[i] + nums[left] + nums[right];
            
            // ===== KEY OPTIMIZATION =====
            if (sum < target) {
                // If sum < target, then:
                // - arr[i] + arr[left] + arr[right] < target ✓
                // - arr[i] + arr[left] + arr[right-1] < target ✓
                // - arr[i] + arr[left] + arr[right-2] < target ✓
                // - ... (all elements between left and right work!)
                // - arr[i] + arr[left] + arr[left+1] < target ✓
                // 
                // That's (right - left) valid triplets!
                // Add them ALL at once
                count += (right - left);
                
                // Move left forward to find more combinations
                left++;
            } else {
                // Sum >= target, need smaller sum
                // Move right backward
                right--;
            }
        }
    }
    
    return count;
}

/*
 * Example Trace: nums = [-2, 0, 1, 3], target = 2
 * After sort: same
 * 
 * i=0, arr[i]=-2
 * [-2, 0, 1, 3]
 *   ^  L     R
 * 
 * sum = -2+0+3 = 1 < 2 ✓
 * 
 * How many valid triplets with arr[i]=-2 and arr[left]=0?
 * - [-2, 0, 3] → sum = 1 < 2 ✓
 * - [-2, 0, 1] → sum = -1 < 2 ✓
 * 
 * Count = right - left = 3 - 1 = 2 ✓
 * This is the KEY optimization!
 * 
 * Without this, we'd need nested loops to check each combination
 * With this, we count all valid combinations in O(1)
 */
