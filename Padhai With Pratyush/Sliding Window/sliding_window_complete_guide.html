🧠 Logic Behind the Solution

Core Strategy: Sliding Window (Fixed Size)

Why this works: Instead of recalculating sum for every subarray from scratch (O(n*k)), we "slide" the window by removing the leftmost element and adding the rightmost element.
Key insight: Most elements stay in the window - only one leaves, one enters!

The Mathematical Trick

When moving from [i...i+k-1] to [i+1...i+k]:
New Sum = Old Sum - arr[i] + arr[i+k]

Example: [1,2,3,4,5], k=3
• Sum[1,2,3] = 6
• Sum[2,3,4] = 6 - 1 + 4 = 9 (instead of recalculating 2+3+4)

Step-by-Step Logic

  1. Calculate first window sum → arr[0] to arr[k-1]
  2. Slide the window: For each new position i from k to n-1:
    • Remove: Subtract arr[i-k] (element leaving)
    • Add: Add arr[i] (element entering)
    • Compare: Update maxSum if current sum is larger

Why O(N) Time?

• We visit each element exactly once
• Each operation (add/subtract) is O(1)
• No nested loops needed!
vs Brute Force: O(n*k) - would recalculate sum for each window 

/**
 * FIXED WINDOW PATTERN
 * Problem: Find maximum sum of subarray with size K
 * 
 * Visual Example: arr = [2, 3, 1, 4, 5], k = 3
 * Step 1: [2, 3, 1] → sum = 6
 * Step 2: [3, 1, 4] → sum = 8 (remove 2, add 4)
 * Step 3: [1, 4, 5] → sum = 10 (remove 3, add 5)
 * 
 * Time Complexity: O(n)
 * Space Complexity: O(1)
 */
public int maxSumSubarray(int[] arr, int k) {
    // Edge case: array smaller than window size
    if (arr.length < k) return -1;
    
    // STEP 1: Calculate first window sum [0 to k-1]
    int sum = 0;
    for (int i = 0; i < k; i++) {
        sum += arr[i];
    }
    
    // Initialize max with first window
    int maxSum = sum;
    int low = 0;        // Left boundary of window
    int high = k;       // Points to next element to include
    
    // STEP 2: Slide the window one position at a time
    while (high < arr.length) {
        // 🔑 KEY INSIGHT: Subtract element leaving, add element entering
        // This gives O(1) per slide instead of O(k)
        sum = sum - arr[low] + arr[high];
        
        // Update maximum if current window is better
        maxSum = Math.max(maxSum, sum);
        
        // Move both pointers forward (maintain window size k)
        low++;
        high++;
    }
    
    return maxSum;
}

Type B: Variable Size Window

Situation: Window size is NOT specified, it expands and shrinks dynamically

🔄 Workplace HR Strategy Analogy:

Expand (high++): "Hire" new elements to achieve your goal
Shrink (low++): "Fire" elements to optimize or maintain validity

  1. Expand high pointer — add element to sum

🧠 Logic Behind the Solution

Why Shrink While Valid?

We want the shortest window, so whenever we find a valid window we immediately try to make it smaller. We keep removing from the left as long as the window stays valid. The moment it becomes invalid, we stop — we've found the shortest version of this valid window.

What Changes Compared to Maximum?

Goal Shrink when Record result
MAXIMUM While INVALID After shrinking (when valid again)
MINIMUM While VALID Before shrinking (while still valid)

Common Mistake

❌ Recording minLen AFTER shrinking — the window is no longer valid at that point!

✅ Record minLen BEFORE shrinking — while the window is still confirmed valid. 

/**
 * MINIMUM WINDOW — PROBLEM 1: Subarray Sum ≥ Target
 *
 * Validity: currentSum >= target
 * Data structure: just an int (running sum) — no map needed
 *
 * nums = [2,3,1,2,4,3], target = 7
 *
 * Expand 2:       [2]       sum=2  < 7
 * Expand 3:       [2,3]     sum=5  < 7
 * Expand 1:       [2,3,1]   sum=6  < 7
 * Expand 2:       [2,3,1,2] sum=8  >= 7 ✓ minLen=4
 *   Shrink 2:     [3,1,2]   sum=6  < 7  stop
 * Expand 4:       [3,1,2,4] sum=10 >= 7 ✓ minLen=4
 *   Shrink 3:     [1,2,4]   sum=7  >= 7 ✓ minLen=3
 *   Shrink 1:     [2,4]     sum=6  < 7  stop
 * Expand 3:       [2,4,3]   sum=9  >= 7 ✓ minLen=3
 *   Shrink 2:     [4,3]     sum=7  >= 7 ✓ minLen=2 🎯
 *
 * Time: O(n), Space: O(1)
 */
public int minSubArrayLen(int target, int[] nums) {
    int low = 0, sum = 0;
    int minLen = Integer.MAX_VALUE;

    for (int high = 0; high < nums.length; high++) {

        // EXPAND: add next element
        sum += nums[high];

        // SHRINK WHILE VALID: sum still meets target
        while (sum >= target) {
            // Record before shrinking — window is valid RIGHT NOW
            minLen = Math.min(minLen, high - low + 1);

            // Try making it smaller
            sum -= nums[low];
            low++;
        }
    }

    return (minLen == Integer.MAX_VALUE) ? 0 : minLen;
}

📋 Variable Window — The Template

Every variable window problem follows the same three steps. Only the information you track and the condition you check ever change.

  1. For each new element, add it to your information (sum, frequency map, count, etc.)
  2. While that information breaks the constraint, remove the leftmost element from your information and move left forward.
  3. Once valid, calculate the window length and update the result if it is the best so far.

Example: nums = [1, 2, 3, 4, 5], K = 7 → Answer: 3 (window [1, 2, 3] or [2, 3] ... longest is [1,2,3] or [3,4])

🧠 Logic Behind the Solution

Why Shrink Only When Invalid?

We want the longest window, so we expand as greedily as possible. We only shrink when we're forced to — when the sum exceeds K. Once valid again, we immediately record the length and go back to expanding.

Critical Detail: While vs If for Shrinking

❌ if (sum > K) — Wrong! Only removes one element. If adding 10 to a sum of 5 with K=7, removing one small element won't fix it.

✅ while (sum > K) — Correct! Keeps removing until the sum is back within K.

Where to Record maxLen

Record after the shrink loop — at that point the window is guaranteed valid. Every valid state is a candidate for the answer. 

/**
 * MAXIMUM WINDOW — PROBLEM 1: Longest Subarray with Sum ≤ K
 *
 * Validity: currentSum <= K
 * Data structure: just an int (running sum) — no map needed
 *
 * nums = [1, 2, 3, 4, 5], K = 7
 *
 * Expand 1: [1]       sum=1  ≤ 7 ✓ maxLen=1
 * Expand 2: [1,2]     sum=3  ≤ 7 ✓ maxLen=2
 * Expand 3: [1,2,3]   sum=6  ≤ 7 ✓ maxLen=3
 * Expand 4: [1,2,3,4] sum=10 > 7 ✗
 *   Shrink 1: [2,3,4] sum=9  > 7 ✗
 *   Shrink 2: [3,4]   sum=7  ≤ 7 ✓ maxLen=3 (no change)
 * Expand 5: [3,4,5]   sum=12 > 7 ✗
 *   Shrink 3: [4,5]   sum=9  > 7 ✗
 *   Shrink 4: [5]     sum=5  ≤ 7 ✓ maxLen=3 (no change)
 *
 * Time: O(n), Space: O(1)
 */
public int longestSubarraySumK(int[] nums, int k) {
    int low = 0, sum = 0, maxLen = 0;

    for (int high = 0; high < nums.length; high++) {

        // EXPAND: add next element
        sum += nums[high];

        // SHRINK WHILE INVALID: sum exceeds K
        while (sum > k) {
            sum -= nums[low];
            low++;
        }

        // Record after shrinking — window is valid now
        maxLen = Math.max(maxLen, high - low + 1);
    }

    return maxLen;
}

🧠 Logic Behind the Solution

Core Strategy: Expand Until Invalid, Then Shrink

Why this works: We want the LONGEST valid window, so we expand greedily and only shrink when forced.
Think of it like filling a container - keep adding until it overflows, then remove from the start just enough to make it valid again.

The Key Insights

🎯 Expand first, shrink only when invalid
📊 HashMap tracks distinct characters - size() = count of unique chars
⚖️ While loop for shrinking - might need multiple removals
📈 Update max continuously - every valid window could be the answer

Why HashMap?

  • map.size() gives us distinct character count instantly
  • map.get(char) tracks frequency for proper removal
  • Remove from map when count reaches 0 (keeps size() accurate)

Decision Flow

  1. Add character → map.put(char, freq + 1)
  2. Check validity → is map.size() > k?
  3. If INVALID: Shrink left until map.size() ≤ k
  4. If VALID: maxLen = max(maxLen, windowSize)
  5. Continue → Move high pointer

Critical Detail: While vs If

❌ if (map.size() > k) - Wrong! Only shrinks once
✅ while (map.size() > k) - Correct! Shrinks until valid

Example: "AAAAB" with k=1 needs to remove 4 A's, not just 1 

/**
 * Longest Substring with K Distinct Characters
 *
 * s = "araaci", k = 2
 *
 * Expand a: freq={a:1}         size=1 ≤ 2 → valid,  res=1
 * Expand r: freq={a:1,r:1}     size=2 ≤ 2 → valid,  res=2
 * Expand a: freq={a:2,r:1}     size=2 ≤ 2 → valid,  res=3
 * Expand a: freq={a:3,r:1}     size=2 ≤ 2 → valid,  res=4  ← exactly k=2 ✓
 * Expand c: freq={a:3,r:1,c:1} size=3 > 2 → INVALID
 *   Shrink a: freq={a:2,r:1,c:1} still 3 → still invalid
 *   Shrink r: freq={a:2,c:1}     size=2 → valid again
 * Expand i: freq={a:2,c:1,i:1} size=3 > 2 → INVALID
 *   Shrink a: freq={a:1,c:1,i:1} still 3 → still invalid
 *   Shrink a: freq={c:1,i:1}     size=2 → valid, but size=2 not exactly k=2... wait it is!
 *
 * res = 4  ("araa")
 *
 * 🔑 Only update res when freq.size() == k (exactly k distinct)
 * Time: O(n)  Space: O(k)
 */
public int longestSubstringKDistinct(String s, int k) {
    int n = s.length();
    int low = 0, res = -1;
    Map<Character, Integer> freq = new HashMap<>();

    for (int high = 0; high < n; high++) {
        char c = s.charAt(high);

        // EXPAND: add character at high into the frequency map
        freq.put(c, freq.getOrDefault(c, 0) + 1);

        // SHRINK: if we now have more than k distinct chars, move low forward
        // Use while — removing one char may not be enough (e.g. "AAAB" with k=1)
        while (freq.size() > k) {
            char leftChar = s.charAt(low);
            freq.put(leftChar, freq.get(leftChar) - 1);

            // Remove the entry entirely when count hits 0
            // — this keeps freq.size() accurate as our distinct-char counter
            if (freq.get(leftChar) == 0)
                freq.remove(leftChar);

            low++;
        }

        // UPDATE: only record when we have EXACTLY k distinct chars
        // (windows with fewer than k don't satisfy the problem)
        if (freq.size() == k)
            res = Math.max(res, high - low + 1);
    }

    return res; // returns -1 if no valid window with exactly k distinct chars found
}

🧠 Code Explanation

Variables

freq — HashMap tracking how many times each character appears in the current window. freq.size() tells us the number of distinct characters.
low — left boundary of the window, advances when we need to shrink.
res = -1 — initialised to -1 (not 0) so we can detect when no valid window was ever found.

Expand step

freq.put(c, freq.getOrDefault(c, 0) + 1)
Adds the new character to the map. If it's already there, increments its count. If it's new, inserts it with count 1. After this, freq.size() may have grown by 1 if it was a new distinct character.

Shrink step

while (freq.size() > k) — keeps shrinking until we're back to k or fewer distinct chars.

Inside: decrement the left character's count, then remove the entry entirely if it hits 0. This is the critical step — if you leave a zero-count entry in the map, freq.size() will be wrong and you'll shrink too much.

Why freq.size() == k and not ≤ k?

The problem asks for exactly k distinct characters. A window with only 1 distinct char when k=2 is not a valid answer. Checking == k ensures we only record windows that fully satisfy the constraint. If the problem said "at most k", you'd use ≤ k instead.

Return value

Returns res, which stays -1 if no window with exactly k distinct characters was ever found (e.g. the string has fewer than k unique characters). This is safer than returning 0, which could be mistaken for an empty valid window.

⚖️ Pattern Comparison

📈 MAXIMUM Pattern Logic

✓ Window is VALID
→ Keep expanding
→ Update max result

✗ Window becomes INVALID
→ Shrink until valid
→ Then continue expanding

Goal: Make window as LARGE as possible while staying valid

📉 MINIMUM Pattern Logic

✗ Window is INVALID
→ Keep expanding
→ Looking for valid window

✓ Window becomes VALID
→ Update min result
→ Shrink to find smaller

Goal: Make window as SMALL as possible while staying valid

Every slide

🧺 Problem 4: Fruits Into Baskets (At Most 2 Distinct)

🎯 Core Concept

You have 2 baskets. Each basket can hold only one type of fruit. Pick from a row of trees, but you must pick continuously. Find the maximum number of fruits you can collect.

Translation: Find the longest subarray with at most 2 distinct values.

Key: "At most 2" → valid when freq.size() ≤ 2 → update maxLen after every valid window.

Input: fruits = [1, 2, 1, 2, 3]
Baskets: can hold type 1 and type 2 — but not type 3
Output: 4 (collect [1, 2, 1, 2])

💻 Java Implementation

/**
 * Fruits Into Baskets
 * = Longest subarray with at most 2 distinct values
 *
 * fruits = [1, 2, 1, 2, 3]
 *
 * right=0: freq={1:1}       size=1 ≤ 2  maxLen=1
 * right=1: freq={1:1,2:1}   size=2 ≤ 2  maxLen=2
 * right=2: freq={1:2,2:1}   size=2 ≤ 2  maxLen=3
 * right=3: freq={1:2,2:2}   size=2 ≤ 2  maxLen=4
 * right=4: freq={1:2,2:2,3:1} size=3 > 2 → shrink
 *   left=0 remove 1: freq={1:1,2:2,3:1} still 3
 *   left=1 remove 2: freq={1:1,2:1,3:1} still 3
 *   left=2 remove 1: freq={2:1,3:1}     size=2 ✓  maxLen=4
 *
 * Time: O(n)  Space: O(1) — freq holds at most 3 entries
 */
public int totalFruit(int[] fruits) {
    int n = fruits.length;
    int left = 0, maxLen = 0;
    Map<Integer, Integer> freq = new HashMap<>();

    for (int right = 0; right < n; right++) {

        // EXPAND: put fruit at right into a basket
        freq.put(fruits[right], freq.getOrDefault(fruits[right], 0) + 1);

        // SHRINK: too many fruit types — empty from the left basket
        while (freq.size() > 2) {
            int leftFruit = fruits[left];
            freq.put(leftFruit, freq.get(leftFruit) - 1);
            // empty basket completely before moving on
            if (freq.get(leftFruit) == 0)
                freq.remove(leftFruit);
            left++;
        }

        // valid window (≤ 2 types) — record if longest so far
        maxLen = Math.max(maxLen, right - left + 1);
    }

    return maxLen;
}

🧠 Code Explanation

Variables

freq — maps each fruit type to how many of that type are in the current window. freq.size() = number of distinct types in the baskets.
left — left boundary; advances when we have more than 2 types.
maxLen — tracks the longest valid window seen so far.

Expand step

freq.put(fruits[right], freq.getOrDefault(fruits[right], 0) + 1)
Picks the fruit at right and adds it to the basket. If this fruit type is already there, increments its count. If it's new, inserts it with count 1 — potentially pushing freq.size() to 3.

Shrink step

while (freq.size() > 2) — keeps emptying the left of the window until we're back to 2 types.

Inside: decrement the left fruit's count. If it hits 0, remove the entry from the map — this is what reduces freq.size() back down. Without the remove, freq.size() would stay at 3 even with a zero-count entry and the loop would never exit.

Why update maxLen outside the while loop?

By the time we reach maxLen = Math.max(...), the while loop has already finished — meaning freq.size() ≤ 2 is guaranteed. The window is valid, so it's safe to record. This is the MAXIMUM pattern: shrink when invalid, record when valid.

At most 2 vs exactly 2

Exactly 2 — window is only valid when freq.size() == 2
At most 2 — window is valid when freq.size() == 2 or freq.size() < 2

/**
 * Problem: Longest Substring Without Repeating Characters
 * Type: Variable Window - Maximum/Longest
 * Pattern: Shrink when INVALID (duplicates exist)
 * Check: map.size() should equal window length
 */
public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> map = new HashMap<>();
    int low = 0, maxLen = 0;
    
    for (int high = 0; high < s.length(); high++) {
        // Expand: Add character
        char c = s.charAt(high);
        map.put(c, map.getOrDefault(c, 0) + 1);
        
        // Shrink WHILE INVALID (duplicates exist)
        // Invalid = map.size() < window_length
        while (map.size() < (high - low + 1)) {
            char leftChar = s.charAt(low);
            map.put(leftChar, map.get(leftChar) - 1);
            if (map.get(leftChar) == 0) {
                map.remove(leftChar);
            }
            low++;
        }
        
        // Update result (all unique now)
        maxLen = Math.max(maxLen, high - low + 1);
    }
    
    return maxLen;
}

 * Time Complexity: O(n) - each character visited at most twice
 * Space Complexity: O(min(n, 26)) - at most 26 lowercase letters
 */

/**
 * Longest Repeating Character Replacement
 *
 * s = "AABABBA", k = 1
 *
 * high=0: freq={A:1}       maxCnt=1, len=1, diff=0 ≤ 1 ✓  res=1
 * high=1: freq={A:2}       maxCnt=2, len=2, diff=0 ≤ 1 ✓  res=2
 * high=2: freq={A:2,B:1}   maxCnt=2, len=3, diff=1 ≤ 1 ✓  res=3
 * high=3: freq={A:3,B:1}   maxCnt=3, len=4, diff=1 ≤ 1 ✓  res=4
 * high=4: freq={A:3,B:2}   maxCnt=3, len=5, diff=2 > 1 ✗
 *   low=0 remove A: freq={A:2,B:2} maxCnt=2, len=4, diff=2 > 1 ✗
 *   low=1 remove A: freq={A:1,B:2} maxCnt=2, len=3, diff=1 ≤ 1 ✓  res=4
 * high=5: freq={A:1,B:3}   maxCnt=3, len=4, diff=1 ≤ 1 ✓  res=4
 * high=6: freq={A:2,B:3}   maxCnt=3, len=5, diff=2 > 1 ✗
 *   low=2 remove B: freq={A:2,B:2} maxCnt=2, len=4, diff=2 > 1 ✗
 *   low=3 remove A: freq={A:1,B:2} maxCnt=2, len=3, diff=1 ≤ 1 ✓  res=4
 *
 * Time: O(n * d) where d = distinct chars in window → O(n)
 * Space: O(d) = O(26) for lowercase letters = O(1)
 */
// Helper: find the highest frequency value in the map
private int findMax(Map<Character, Integer> freq) {
    int maxc = -1;
    for (int count : freq.values())
        maxc = Math.max(maxc, count);
    return maxc;
}

public int characterReplacement(String s, int k) {
    int n = s.length();
    Map<Character, Integer> freq = new HashMap<>();
    int low = 0, res = Integer.MIN_VALUE;

    for (int high = 0; high < n; high++) {
        char c = s.charAt(high);

        // Step 1: add high to information
        freq.put(c, freq.getOrDefault(c, 0) + 1);

        // Step 2: while invalid, shrink from left
        // Valid   when: diff = len - maxCnt ≤ k  (replacements needed ≤ budget)
        // Invalid when: diff = len - maxCnt > k  (would need more replacements than allowed)
        // Why? len - maxCnt = number of non-dominant chars = chars that must be replaced
        // e.g. window [AABAB]: len=5, maxCnt=3(A) → 5-3=2 replacements needed, k=1 → INVALID
        int maxCnt = findMax(freq);       // most frequent char in window
        int len    = high - low + 1;
        int diff   = len - maxCnt;        // number of chars that need replacing
        while (diff > k) {
            char leftChar = s.charAt(low);
            freq.put(leftChar, freq.get(leftChar) - 1);
            if (freq.get(leftChar) == 0)
                freq.remove(leftChar);  // keep map clean so values() stays accurate
            low++;
            maxCnt = findMax(freq);
            len    = high - low + 1;
            diff   = len - maxCnt;
        }

        // Step 3: window is valid — record result
        len = high - low + 1;
        res = Math.max(res, len);
    }

    return res;
}

📉 MINIMUM Window — Problem 2: Smallest Window Containing All Characters

Problem: Given strings S and T, find the smallest substring of S that contains every character in T (including duplicates).

Example: S = "ADOBECODEBANC", T = "ABC" → Answer: "BANC"

Validity condition: have[c] ≥ need[c] for every character in T
Data structure needed: two frequency arrays — one for what T requires, one for what the current window has

⚠️ Notice the difference from Problem 1

Problem 1 checks a single number: sum ≥ target — one comparison.
Problem 2 checks across every character: have[c] ≥ need[c] for all c — needs a helper function.

The shrink-while-valid pattern is identical. Only the validity check is different.

Core Logic: Same pattern as Problem 1 — shrink WHILE valid
  1. Build need[] from T — how many of each character we require
  2. Expand high pointer — add character to have[]
  3. Check validity: does have[] satisfy need[] for every character?
  4. While VALID: record current window, then shrink from left to try for smaller

Key Pattern: Identical shrink-while-valid logic as Problem 1. Only the validity check changed — from sum ≥ target to isValid(have, need).

🧠 Logic Behind the Solution

The Validity Check

A window is valid when it has enough of every character T requires. We track this with two arrays:
• need[c] = how many times character c appears in T
• have[c] = how many times c is currently in our window
Valid when have[c] ≥ need[c] for every c.

Why a Helper Function?

Problem 1: validity = one number comparison → inline check
Problem 2: validity = 256 character comparisons → extract to isValid(have, need)

The helper loops over all characters and returns false the moment any character is under-supplied. Same logic, just more complex to express inline.

Common Mistake

❌ Recording the window AFTER shrinking — the window is invalid at that point.

✅ Record startInd and minLen BEFORE removing from the left — while the window is still valid.

The Pattern Stays the Same

Problem 1 (Sum) Problem 2 (Substring)
Expand step sum += nums[high] have[s[high]]++
Valid check sum ≥ target isValid(have, need)
Record result minLen = min(minLen, size) minLen = size; startInd = low
Shrink step sum -= nums[low]; low++ have[s[low]]--; low++ 
/**
 * MINIMUM WINDOW — PROBLEM 2: Smallest Substring Containing All of T
 *
 * Validity: have[c] >= need[c] for every character c in T
 * Data structure: two int[256] frequency arrays
 *
 * S = "ADOBECODEBANC", T = "ABC"
 * need = {A:1, B:1, C:1}
 *
 * Expand to "ADOBEC"  → valid (has A,B,C) → record len=6
 *   Shrink: "DOBEC"   → invalid (no A) → stop
 * Expand to "DOBECODEB A NC" → valid → len=12, no improvement
 *   Shrink down to "BANC" → valid → record len=4 🎯
 *   Shrink: "ANC"     → invalid (no B) → stop
 *
 * Time: O(|S| + |T|), Space: O(1) — fixed 256-char arrays
 */
public String minWindow(String s, String t) {
    if (s.length() == 0 || t.length() == 0) return "";

    int[] need = new int[256]; // what T requires
    int[] have = new int[256]; // what current window contains

    // Build need[] from T
    for (char c : t.toCharArray()) need[c]++;

    int low = 0, minLen = Integer.MAX_VALUE, startInd = -1;

    for (int high = 0; high < s.length(); high++) {

        // EXPAND: add character to window
        have[s.charAt(high)]++;

        // SHRINK WHILE VALID: window satisfies all of T's requirements
        while (isValid(have, need)) {
            // Record before shrinking — window is valid RIGHT NOW
            if (high - low + 1 < minLen) {
                minLen = high - low + 1;
                startInd = low;
            }

            // Try making it smaller
            have[s.charAt(low)]--;
            low++;
        }
    }

    return minLen == Integer.MAX_VALUE ? "" : s.substring(startInd, startInd + minLen);
}

// Valid when current window has enough of every required character
private boolean isValid(int[] have, int[] need) {
    for (int i = 0; i < 256; i++) {
        if (have[i] < need[i]) return false;
    }
    return true;
}

🧠 Logic Behind the Solution

The Challenge

Most complex sliding window problem!
• Must contain ALL characters from T (including duplicates)
• Want MINIMUM length window
• Characters in S can be in any order
• T can have repeated characters (e.g., "AAB")

Two HashMap Strategy

HashMap 1 (targetMap): Frequency of each char in T
• Example: T = "AAB" → {A:2, B:1}
• This never changes

HashMap 2 (windowMap): Frequency of chars in current window
• Updates as we expand/shrink window
• Example: Window = "BA" → {B:1, A:1}

formed variable: Count of characters that have met their requirement
• If T needs 2 A's and we have 2 A's → formed++
• Need formed == targetMap.size() for valid window

The "formed" Logic Explained

Example: T = "AAB"
• targetMap = {A:2, B:1}
• required = 2 (need both 'A' and 'B' satisfied)

Window processing:
• Add 'A': windowMap={A:1}, formed=0 (need 2 A's)
• Add 'A': windowMap={A:2}, formed=1 ✓ (A satisfied)
• Add 'B': windowMap={A:2,B:1}, formed=2 ✓ (Both satisfied)
• Now valid! formed == required

Expand and Shrink Logic

  1. EXPAND (right pointer):
    • Add character to windowMap
    • If this char is in T and frequencies now match → formed++
  2. CHECK VALIDITY: Is formed == required?
  3. SHRINK (left pointer) - while valid:
    • Update result if current window is smaller
    • Remove leftmost character
    • If removed char was required and now insufficient → formed--

Complete Example

S = "ADOBECODEBANC", T = "ABC"
targetMap = {A:1, B:1, C:1}, required = 3

Process:
• [A,D,O,B,E,C] → formed=3 ✓ First valid! minLen=6
• Shrink: [D,O,B,E,C] → formed=2 (lost A), invalid
• Expand: [D,O,B,E,C,O,D,E,B,A] → formed=3 ✓
• Shrink: [B,E,C,O,D,E,B,A] → ...
• Shrink: [B,A,N,C] → formed=3 ✓, len=4 (NEW MIN!)
• Shrink: [A,N,C] → formed=2 (lost B), stop
Answer: "BANC" (length 4)

Key Insights

  • Two pointers never backtrack → O(N + M) time
  • formed variable provides O(1) validity check
  • Shrink while valid (opposite of maximum problems)
  • Update result BEFORE shrinking (window is valid NOW)

// Use when: Window size K is given
public int fixedWindowTemplate(int[] arr, int k) {
    int windowSum = 0;
    int result = 0; // or Integer.MIN_VALUE/MAX_VALUE
    
    // Calculate first window
    for (int i = 0; i < k; i++) {
        windowSum += arr[i];
    }
    result = windowSum;
    
    // Slide window
    for (int i = k; i < arr.length; i++) {
        windowSum = windowSum - arr[i - k] + arr[i];
        result = Math.max(result, windowSum); // or min, or other operation
    }
    
    return result;
}

// Use when: Finding shortest/minimum with constraint "at least K" or "≥ target"
public int variableMinTemplate(int[] arr, int target) {
    int low = 0, sum = 0;
    int minResult = Integer.MAX_VALUE;
    
    for (int high = 0; high < arr.length; high++) {
        // 1. EXPAND: Add arr[high] to window
        sum += arr[high];
        
        // 2. SHRINK: While window is VALID (condition satisfied)
        while (/* window is valid */) {
            // 3. UPDATE RESULT: Record this valid window
            minResult = Math.min(minResult, high - low + 1);
            
            // Try to shrink further
            sum -= arr[low];
            low++;
        }
    }
    
    return minResult == Integer.MAX_VALUE ? 0 : minResult;
}

// - while (sum >= target)           // Sum meets requirement
