📚 Part 1: LinkedList Fundamentals

What is a LinkedList?

A LinkedList is a linear data structure where elements (nodes) are stored at non-contiguous memory locations. Each node contains:
  • Data: The actual value stored
  • Next pointer: Memory address of the next node
Unlike arrays, LinkedLists don't need contiguous memory and can grow/shrink dynamically!

🎯 PATTERN 1: Fast & Slow Pointers (Floyd's Tortoise & Hare)

🔍 How to Recognize This Pattern?

You should think "Fast & Slow Pointers" when you see:

✓ "Find the middle" of the list
✓ "Detect a cycle" in the list
✓ "Find the kth node from the end"
✓ "Check if palindrome" (needs middle + reverse)
✓ "Find the start of a cycle"
✓ Any question where you need to find a position without counting

Keywords to watch for:
"cycle", "loop", "middle", "kth from end", "palindrome", "meet", "intersection"

💡 The Intuition - Why This Pattern Works

The Race Track Analogy:

Imagine two runners on a circular track:
🐢 Slow runner: Moves 1 step per turn
🐇 Fast runner: Moves 2 steps per turn

Case 1: No cycle (linear track):
• Fast reaches the end (null) first
• They never meet

Case 2: Cycle exists (circular track):
• Fast enters cycle first
• Fast is now ahead of slow in the cycle
• Each iteration, fast gains 1 position on slow
• Eventually they MUST meet (like lapping on a track!)🏃

Finding Middle:
• When fast reaches end (moved 2n steps)
• Slow has moved n steps → at middle!

Why it works: Fast moves twice as fast, so when fast is done, slow is halfway!

📋 Why This Pattern Applies

Problem: Does a cycle exist in the list?
Why Fast & Slow? If cycle exists, fast will eventually "lap" slow and they'll meet!
Alternative approaches: HashSet (O(n) space) - But this is O(1) space!

🎯 PATTERN 2: Reverse LinkedList

🔍 How to Recognize This Pattern?

You should think "Reverse LinkedList" when you see:

✓ "Reverse the entire list"
✓ "Reverse between positions m and n"
✓ "Reverse in k-groups" (reverse every k nodes)
✓ "Swap nodes in pairs"
✓ "Check if palindrome" (needs reverse of half)
✓ "Reorder list" (L0→Ln→L1→Ln-1...)

Keywords to watch for:
"reverse", "backward", "swap pairs", "palindrome", "reorder", "flip"

💡 The Intuition - Why This Pattern Works

The Core Challenge:

To reverse 1→2→3→NULL into 3→2→1→NULL

🎯 What we're actually doing: Reversing the arrows!
• Change 1→2 into 1←2
• Change 2→3 into 2←3
• Change 3→NULL into 3←NULL (wait, that becomes NULL←3)

The Problem:
If we just do: curr.next = prev to reverse 1→2 to 1←2,
we LOSE access to node 3! 😱

The Solution: 3 Pointers!
• prev: The node that curr should point to (what's behind)
• curr: Current node we're reversing
• next: Save the next node BEFORE we reverse!

Process (The Dance):
1. Save next node: next = curr.next
2. Reverse arrow: curr.next = prev
3. Move prev forward: prev = curr
4. Move curr forward: curr = next
5. Repeat until curr is null
6. Return prev (new head!)

📋 Why This Pattern Applies

Problem: Reverse 1→2→3→NULL to 3→2→1→NULL
Why 3 Pointers? Need to track: where we came from (prev), where we are (curr), where we're going (next)
Alternative: Use stack (O(n) space) or recursion (O(n) space) - But iterative is O(1) space!
Key Insight: We're changing pointers, not moving data!

// Pattern 2: Reverse LinkedList - Iterative public ListNode reverseList(ListNode head) { ListNode prev = null; ListNode curr = head; while (curr != null) { ListNode next = curr.next; // Save next (don't lose it!) curr.next = prev; // Reverse the arrow prev = curr; // Move prev forward curr = next; // Move curr forward } return prev; // New head! } // Time: O(n) - Visit each node once // Space: O(1) - Only 3 pointers

// Variation: Reverse Between Positions m and n public ListNode reverseBetween(ListNode head, int m, int n) { if (head == null || m == n) return head; ListNode dummy = new ListNode(0); dummy.next = head; ListNode beforeM = dummy; // Move to node before position m for (int i = 1; i < m; i++) { beforeM = beforeM.next; } // Reverse from m to n ListNode prev = null; ListNode curr = beforeM.next; ListNode tail = curr; // Will become tail of reversed section for (int i = m; i <= n; i++) { ListNode next = curr.next; curr.next = prev; prev = curr; curr = next; } // Connect the pieces beforeM.next = prev; // Connect to new head of reversed section tail.next = curr; // Connect to rest of list return dummy.next; } // Time: O(n), Space: O(1)

🎯 PATTERN 3: Merge Sorted Lists

🔍 How to Recognize This Pattern?

You should think "Merge Lists" when you see:

✓ "Merge two sorted lists"
✓ "Merge k sorted lists"
✓ "Sort a linked list" (use merge sort!)
✓ "Add two numbers" (represented as lists)
✓ Problem involves combining multiple lists

Keywords to watch for:
"merge", "combine", "sort", "sorted lists", "add", "k lists"

💡 The Intuition - Why This Pattern Works

The Merge Sort Concept:

Given two sorted lists:
List1: 1→3→5→NULL
List2: 2→4→6→NULL

🎯 Goal: Create merged sorted list: 1→2→3→4→5→6→NULL

The Process (The Zipper):
1. Compare heads: 1 vs 2 → Pick 1 (smaller)
2. Attach 1 to result, move List1 forward
3. Compare: 3 vs 2 → Pick 2 (smaller)
4. Attach 2 to result, move List2 forward
5. Repeat until one list is empty
6. Attach remaining list

Key Insight: We don't create new nodes!
We just rearrange pointers → O(1) space (excluding output)

Why it works: Both lists are sorted, so we only need to compare current heads!

📋 Why This Pattern Applies

Problem: Merge 1→4 and 2→3 into sorted list
Why Merge Pattern? Both are sorted! Just compare and link!
Alternative: Put all in array, sort, rebuild list - O(n log n) time and O(n) space
This approach: O(n+m) time, O(1) space - Just pointer manipulation!

// Pattern 3: Merge Two Sorted Lists public ListNode mergeTwoLists(ListNode l1, ListNode l2) { // Dummy node simplifies edge cases ListNode dummy = new ListNode(0); ListNode curr = dummy; while (l1 != null && l2 != null) { if (l1.val < l2.val) { curr.next = l1; // Attach l1 node l1 = l1.next; // Move l1 forward } else { curr.next = l2; // Attach l2 node l2 = l2.next; // Move l2 forward } curr = curr.next; // Move curr forward } // Attach remaining nodes (one list is empty) curr.next = (l1 != null) ? l1 : l2; return dummy.next; // Skip dummy node } // Time: O(n + m) - Visit each node once // Space: O(1) - Only pointers, no new nodes

🎯 PATTERN 4: Dummy Node Technique

🔍 How to Recognize This Pattern?

You should think "Dummy Node" when you see:

✓ "Remove nodes" (especially first node)
✓ "Delete elements" from list
✓ "Partition list" into two parts
✓ "Add/merge/build new list"
✓ Anytime head might change during operation
✓ Need to handle edge case where head is affected

Keywords to watch for:
"remove", "delete", "partition", "split", "build", modifying head

💡 The Intuition - Why Dummy Nodes Are Magic

The Problem Without Dummy:

When modifying a list, the HEAD can change!
Example: Remove first node → head = head.next (special case!)
Example: Remove all 1s from 1→2→1→3 → head changes!

🎯 The Magic Solution: Dummy Node!

Create a fake node BEFORE the real head:
dummy(0) → 1 → 2 → 3

Benefits:
1. Head becomes just another node (no special case!)
2. Always have a node before the one we're modifying
3. Can use same code for all nodes
4. Return dummy.next as final result

Think of it as: A handle to hold the list while we modify it!

📋 Why This Pattern Applies

Problem: Remove all nodes with value 1
Why Dummy? If first node is 1, we need to change head - but with dummy, it's just another removal!
Alternative: Check if head needs removal separately, then loop - messy code!
With dummy: Clean, uniform code for all nodes!

// Pattern 4: Dummy Node - Remove Elements public ListNode removeElements(ListNode head, int val) { ListNode dummy = new ListNode(0); dummy.next = head; ListNode curr = dummy; while (curr.next != null) { if (curr.next.val == val) { curr.next = curr.next.next; // Skip the node } else { curr = curr.next; // Move forward } } return dummy.next; // New head (might be different!) } // Time: O(n), Space: O(1)

🎯 PATTERN 5 & 6: In-Place Manipulation + Runner Technique

🔍 How to Recognize These Patterns?

In-Place Manipulation when you see:
✓ "Reorder" the list
✓ "Remove duplicates"
✓ "Odd/even" positions
✓ "Partition" around value
✓ Must modify without extra space

Runner Technique when you see:
✓ "Remove nth from end"
✓ "Find intersection" of two lists
✓ Need two pointers with offset
✓ "Rotate list" by k positions

💡 The Intuition

In-Place Manipulation:
Change pointers without creating new nodes → O(1) space!
Example: Odd-Even → Collect odds, then evens, link together

Runner (Two Pointers with Offset):
Start one pointer ahead by k steps
When leader reaches end, follower is k from end!

Example: Remove 2nd from end in 1→2→3→4→5
• Start fast 2 steps ahead
• Move both until fast reaches end
• Slow is now at node before target!

📋 Why These Patterns Apply

Remove Nth from End - Why Runner?
Don't know length! With 2 pointers offset by n, when first reaches end, second is at target!

Odd-Even List - Why In-Place?
Could use array O(n) space - But can rearrange pointers O(1) space!

// Pattern 6: Runner - Remove Nth From End public ListNode removeNthFromEnd(ListNode head, int n) { ListNode dummy = new ListNode(0); dummy.next = head; ListNode slow = dummy; ListNode fast = dummy; // Move fast n+1 steps ahead for (int i = 0; i <= n; i++) { fast = fast.next; } // Move both until fast reaches end while (fast != null) { slow = slow.next; fast = fast.next; } // Remove the node slow.next = slow.next.next; return dummy.next; } // Time: O(n), Space: O(1)

🎯 PATTERN 7: Recursion

🔍 How to Recognize This Pattern?

You should think "Recursion" when you see:

✓ "Reverse" (can be done recursively)
✓ "Merge" (recursive divide & conquer)
✓ "Swap pairs"
✓ Problem has recursive structure
✓ Question says "try recursive solution"
✓ Need to process from the end backward

Key insight: List IS recursive: head + (rest of list)

💡 The Intuition - Recursive Thinking

The Recursive View:

A LinkedList is: head → (smaller list)

Example: 1→2→3→NULL is:
• Node 1 + (2→3→NULL)
• Which is: Node 1 + (Node 2 + (3→NULL))
• Which is: Node 1 + (Node 2 + (Node 3 + NULL))

🎯 Recursive Pattern:
1. Base case: If head is null (or head.next is null)
2. Recursive step: Do something with head
3. Recurse on rest: Call function on head.next
4. Combine results

Think from the END:
Process last node first, then second-to-last, etc.
Trust recursion to handle "rest of list"!
