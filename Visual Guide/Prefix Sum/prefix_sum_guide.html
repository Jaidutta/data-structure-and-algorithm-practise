📊 What is Prefix Sum?

Definition: A prefix sum (or cumulative sum) is a technique where we precompute the sum of all elements from the start of an array up to each index.

prefix[i] = arr[0] + arr[1] + ... + arr[i]

🎯 When to Use Prefix Sum?

✅ Look for these keywords:

  • "Sum of subarray" - Direct indication
  • "Range sum queries" - Multiple queries on same array
  • "Continuous subarray" - Contiguous elements
  • "Multiple queries" - Preprocessing helps
  • "Count subarrays with sum = K" - HashMap + Prefix
  • "Equilibrium index" - Left sum = Right sum
  • "Product of array except self" - Prefix product variant

⚠️ Don't confuse with:

  • Sliding Window: When you need EXACTLY k elements or min/max of fixed-size window
  • Kadane's Algorithm: When finding MAXIMUM sum subarray (not sum = K)
  • Two Pointers: When sorting is allowed and you need to find pairs
  • Binary Search: When array is sorted and you search for elements

Pattern 1: Basic Range Sum Query

Problem: Given Q queries asking for sum from index L to R

Key Insight: sum(L, R) = prefix[R] - prefix[L-1]

Time: O(N) preprocessing + O(1) per query

🤔 Why Use Prefix Sum for This Pattern?

The Problem Without Prefix Sum:

If we have to answer Q range sum queries on an array, the naive approach would be:

// For each query [L, R], loop through the range
for (int i = L; i <= R; i++) {
    sum += arr[i];  // O(N) per query
}
// Total: O(Q * N) - very slow for large Q!

With 1000 queries on an array of 10,000 elements:

  • ❌ Naive: 1,000 × 10,000 = 10,000,000 operations
  • ✅ Prefix Sum: 10,000 (build) + 1,000 (queries) = 11,000 operations
  • 🚀 Speed up: ~909x faster!

The Key Realization:

Instead of recalculating sums repeatedly, we precompute once and query instantly. This is the fundamental trade-off: spend O(N) time upfront to make every query O(1).

Mathematical Insight:

The sum from L to R is just "total sum up to R" minus "total sum up to L-1":

sum(L, R) = sum(0, R) - sum(0, L-1)

Since we've already computed both prefix[R] and prefix[L-1], this subtraction takes O(1)!

Example: Find sum from index 2 to 4

Array: [3, 1, 4, 2, 5]

Prefix: [3, 4, 8, 10, 15]

Sum(2, 4) = prefix[4] - prefix[1] = 15 - 4 = 11

Verification: arr[2] + arr[3] + arr[4] = 4 + 2 + 5 = 11 ✓

class RangeSumQuery {
    private int[] prefix;
    
    // Constructor: O(N) time
    public RangeSumQuery(int[] arr) {
        prefix = new int[arr.length];
        prefix[0] = arr[0];
        
        // Build prefix sum array
        for (int i = 1; i < arr.length; i++) {
            prefix[i] = prefix[i - 1] + arr[i];
        }
    }
    
    // Query: O(1) time
    public int rangeSum(int left, int right) {
        if (left == 0) {
            return prefix[right];
        }
        return prefix[right] - prefix[left - 1];
    }
}

Pattern 2: Count Subarrays with Sum = K (HashMap)

Problem: Count how many subarrays have sum exactly equal to K

Key Insight: If prefix[j] - prefix[i] = K, then prefix[i] = prefix[j] - K

Time: O(N), Space: O(N)

🤔 Why Use Prefix Sum for This Pattern?

The Problem Without Prefix Sum:

Finding all subarrays with sum = K naively requires checking every possible subarray:

for (int i = 0; i < n; i++) {
    int sum = 0;
    for (int j = i; j < n; j++) {
        sum += arr[j];
        if (sum == k) count++;
    }
}
// O(N²) - checks all N*(N+1)/2 subarrays

For an array of 10,000 elements:

  • ❌ Naive: ~50,000,000 operations (N²)
  • ✅ Prefix Sum + HashMap: ~10,000 operations (N)
  • 🚀 Speed up: ~5000x faster!

The Brilliant Insight:

A subarray from index i to j has sum K if:

prefix[j] - prefix[i-1] = K

Rearranging:

prefix[i-1] = prefix[j] - K

This transforms the problem!

Instead of checking all subarrays, we ask: "As I build prefix sums, have I seen (currentSum - K) before?"

1 At each position j, we know prefix[j] (current cumulative sum)
2 We calculate what prefix[i-1] MUST be: (prefix[j] - K)
3 If we've seen that value before, it means we found valid subarray(s)!
4 HashMap tracks how many times each prefix sum appeared

Why HashMap is Essential:

Multiple positions can have the same prefix sum (especially with negative numbers or zeros). Each occurrence represents a different valid subarray ending at current position!

Example: arr = [1, -1, 1, -1, 1], k = 0

When we reach the end, prefix sum = 1. We look for (1 - 0) = 1 in our map. If the value 1 appeared 3 times before, we found 3 subarrays with sum = 0!

Why HashMap?

1 As we build prefix sum, we check: "Have I seen (currentSum - K) before?"
2 If yes, that means there's a subarray ending at current index with sum = K
3 Store frequency of each prefix sum in HashMap 
public int subarraySum(int[] arr, int k) {
    Map map = new HashMap<>();
    int count = 0;
    int sum = 0;
    
    // Important: Initialize with sum 0 appearing once
    map.put(0, 1);
    
    for (int num : arr) {
        sum += num;  // Current prefix sum
        
        // Check if (sum - k) exists in map
        if (map.containsKey(sum - k)) {
            count += map.get(sum - k);
        }
        
        // Add current sum to map
        map.put(sum, map.getOrDefault(sum, 0) + 1);
    }
    
    return count;
}

Pattern 3: 2D Prefix Sum (Matrix)

Problem: Find sum of any rectangle in a 2D matrix

Key Insight: Build 2D prefix sum, use inclusion-exclusion principle

Time: O(M*N) preprocessing + O(1) per query

🤔 Why Use Prefix Sum for This Pattern?

The Problem Without Prefix Sum:

Finding sum of a rectangle from (r1,c1) to (r2,c2) requires nested loops:

int sum = 0;
for (int i = r1; i <= r2; i++) {
    for (int j = c1; j <= c2; j++) {
        sum += matrix[i][j];
    }
}
// O(M * N) per query - very slow!

For a 1000×1000 matrix with 100 queries:

  • ❌ Naive (worst case): 100 × 1,000,000 = 100,000,000 operations
  • ✅ 2D Prefix Sum: 1,000,000 (build) + 100 (queries) = 1,000,100 operations
  • 🚀 Speed up: ~100x faster!

The 2D Extension:

Just like 1D prefix sum lets us find range sums in O(1), 2D prefix sum lets us find rectangle sums in O(1)!

Building 2D Prefix Sum:

For each cell (i,j), we want the sum of all cells from (0,0) to (i,j):

prefix[i][j] = sum of rectangle from (0,0) to (i,j)

To calculate this, we use:

1 Add rectangle above: prefix[i-1][j]
2 Add rectangle to left: prefix[i][j-1]
3 Subtract overlap (counted twice): prefix[i-1][j-1]
4 Add current cell: matrix[i][j]

Querying Rectangle Sum:

To find sum from (r1,c1) to (r2,c2), we use inclusion-exclusion:

1 Start with entire rectangle to (r2,c2): prefix[r2][c2]
2 Remove rectangle above r1: - prefix[r1-1][c2]
3 Remove rectangle left of c1: - prefix[r2][c1-1]
4 Add back corner (removed twice): + prefix[r1-1][c1-1]

Why the Extra Row/Column?

We use prefix[M+1][N+1] instead of prefix[M][N] to avoid boundary checks. When r1=0 or c1=0, we don't need special cases - the padded row/column (all zeros) handles it automatically!

Real-world Applications:

  • 🖼️ Image processing (sum of pixel values in regions)
  • 🎮 Game development (terrain analysis, area effects)
  • 📊 Data analytics (sum of sales in geographic regions)
  • 🤖 Computer vision (integral images for feature detection)

2D Prefix Sum Formula:

prefix[i][j] = prefix[i-1][j] + prefix[i][j-1] - prefix[i-1][j-1] + matrix[i][j]

Range Sum Formula:

sum(r1,c1,r2,c2) = prefix[r2][c2] - prefix[r1-1][c2] - prefix[r2][c1-1] + prefix[r1-1][c1-1] 
class MatrixRangeSum {
    private int[][] prefix;
    
    public MatrixRangeSum(int[][] matrix) {
        if (matrix.length == 0) return;
        
        int m = matrix.length;
        int n = matrix[0].length;
        
        // Extra row and column to handle boundaries
        prefix = new int[m + 1][n + 1];
        
        for (int i = 1; i <= m; i++) {
            for (int j = 1; j <= n; j++) {
                prefix[i][j] = prefix[i-1][j] + prefix[i][j-1] 
                             - prefix[i-1][j-1] + matrix[i-1][j-1];
            }
        }
    }
    
    public int sumRegion(int r1, int c1, int r2, int c2) {
        // Adjust for 1-indexed prefix array
        r1++; c1++; r2++; c2++;
        
        return prefix[r2][c2] - prefix[r1-1][c2] 
               - prefix[r2][c1-1] + prefix[r1-1][c1-1];
    }
}

Query Example: Sum of rectangle (1,1) to (2,2)

This rectangle in the original matrix:

3 0 1 4
5 6 3 2
1 2 0 1

Using the formula:

1 Total rectangle to (2,2): prefix[3][3] = 21
2 Subtract top: prefix[0][3] = 0
3 Subtract left: prefix[3][0] = 0
4 Add back diagonal (subtracted twice): prefix[0][0] = 0
sum = 21 - 0 - 0 + 0 = 21 ❌ (This is wrong! We want only the yellow area)

Adjusting indices: r1=2, c1=2, r2=3, c2=3 (because 1-indexed)

1 Total to bottom-right: prefix[3][3] = 21
2 Subtract above rectangle: prefix[1][3] = 8
3 Subtract left rectangle: prefix[3][1] = 9
4 Add back overlap: prefix[1][1] = 3
sum = 21 - 8 - 9 + 3 = 11 ✓

Pattern 4: Prefix Product (Product Variant)

Problem: Product of all elements except current

Key Insight: Use prefix product from left and right

Time: O(N), Space: O(1) excluding output

🤔 Why Use Prefix Sum/Product for This Pattern?

The Problem Without Prefix Product:

Computing product of all elements except current one naively:

for (int i = 0; i < n; i++) {
    int product = 1;
    for (int j = 0; j < n; j++) {
        if (i != j) product *= arr[j];
    }
    result[i] = product;
}
// O(N²) - multiply N-1 elements for each position

For an array of 10,000 elements:

  • ❌ Naive: ~100,000,000 multiplications (N²)
  • ✅ Prefix Product: ~30,000 multiplications (3N)
  • 🚀 Speed up: ~3333x faster!

The Key Observation:

For any index i, the "product except self" is:

result[i] = (product of all elements LEFT of i) × (product of all elements RIGHT of i)

Breaking It Down:

Instead of multiplying all elements except one (which requires checking each time), we:

1 First pass (LEFT to RIGHT): Store product of everything to the LEFT
2 Second pass (RIGHT to LEFT): Multiply by product of everything to the RIGHT

Visual Example:

Array: [2, 3, 4, 5]

For index 2 (value 4):

  • Left product: 2 × 3 = 6
  • Right product: 5 = 5
  • Result: 6 × 5 = 30 ✓ (which is 2 × 3 × 5)

Why Not Just Divide?

You might think: "Calculate total product, then divide by current element"

// Why this DOESN'T work:
int totalProduct = 1;
for (int num : arr) totalProduct *= num;
for (int i = 0; i < n; i++) {
    result[i] = totalProduct / arr[i];  // ❌ Problem!
}

Issues with division approach:

  • ❌ Zeros: If arr[i] = 0, division fails (divide by zero)
  • ❌ Problem constraint: Many problems explicitly say "no division allowed"
  • ❌ Integer overflow: Total product might exceed max value

Space Optimization Trick:

We don't need separate arrays for left and right products! We reuse the result array:

  • First pass: result[i] stores left product
  • Second pass: result[i] *= right product
  • Achieves O(1) extra space (output doesn't count)

This same pattern applies to:

  • Finding maximum from both sides (Trapping Rain Water)
  • Finding minimum from both sides
  • Any operation that needs "everything except current"
public int[] productExceptSelf(int[] nums) {
    int n = nums.length;
    int[] result = new int[n];
    
    // Build prefix product from left
    result[0] = 1;
    for (int i = 1; i < n; i++) {
        result[i] = result[i - 1] * nums[i - 1];
    }
    
    // Build prefix product from right and multiply
    int rightProduct = 1;
    for (int i = n - 1; i >= 0; i--) {
        result[i] *= rightProduct;
        rightProduct *= nums[i];
    }
    
    return result;
}

Pattern 5: Running Sum with State

Problem: Track cumulative state (balance, score, etc.)

Key Insight: Modify prefix sum to track specific state transitions

🤔 Why Use Prefix Sum for This Pattern?

The Conceptual Shift:

Prefix sum isn't just about numbers - it's about cumulative tracking of any state!

Traditional Prefix Sum:

prefix[i] = prefix[i-1] + arr[i]

Tracks: cumulative sum of values

Generalized Prefix State:

state[i] = f(state[i-1], arr[i])

Tracks: cumulative state based on any function f

Real-World Examples:

1 Bank Balance Problem:
Track deposits (+) and withdrawals (-)
State = running balance = previous balance + transaction
2 Parentheses Matching:
'(' → +1, ')' → -1
State = nesting depth = previous depth + current change
3 Binary Array (0s and 1s):
Treat 0 as -1, keep 1 as +1
State = 0 when equal number of 0s and 1s
4 Score Tracking:
Win (+1), Loss (-1), Draw (0)
State = cumulative score

The Power of State Mapping:

By mapping elements to +1/-1/0 or other values, we transform complex problems into prefix sum problems!

Example: Contiguous Array (LC 525)

Problem: Find longest subarray with equal 0s and 1s

Trick: Convert 0 → -1, keep 1 → +1

Original:  [0, 1, 0, 1, 1, 0]
Converted: [-1, 1, -1, 1, 1, -1]
Prefix:    [-1, 0, -1, 0, 1, 0]

When prefix sum returns to same value, it means equal 0s and 1s in between!

  • Prefix = 0 at index 1 and index 3 → subarray [1,0,1] has equal 0s and 1s ✓
  • Prefix = 0 at index 1 and index 5 → subarray [1,0,1,1,0] has equal 0s and 1s ✓

Why This Pattern Works:

1 Transform problem: Map original values to increments/decrements
2 Track cumulative state: Use prefix sum on transformed values
3 Use HashMap: Store first occurrence of each state
4 Find answer: When state repeats, you've found a valid range

Common State Transformations:

Problem Type Transformation What State = 0 Means
Equal 0s and 1s 0 → -1, 1 → +1 Same count of both
Equal vowels/consonants vowel → +1, consonant → -1 Equal counts
Balanced parentheses '(' → +1, ')' → -1 All matched
Win/Loss balance win → +1, loss → -1 Even record

Key Insight:

Prefix sum isn't just addition - it's about tracking how we got to the current state. If two positions have the same state, everything in between "canceled out" or "balanced"!

// Example: Check if parentheses are balanced
public boolean isValid(String s) {
    int balance = 0;
    
    for (char c : s.toCharArray()) {
        if (c == '(') balance++;
        else balance--;
        
        // Never go negative (more closing than opening)
        if (balance < 0) return false;
    }
    
    return balance == 0;
}

Pattern 6: Difference Array (Range Updates)

Problem: Multiple range updates, then query final state

Key Insight: Mark start (+val) and end+1 (-val), then prefix sum

Time: O(N + Q) instead of O(N*Q)

🤔 Why Use Prefix Sum for This Pattern?

The Problem Without Difference Array:

When you have multiple range updates to apply to an array:

// Naive approach: Update every element in range
for (Update update : updates) {
    for (int i = update.start; i <= update.end; i++) {
        arr[i] += update.value;
    }
}
// O(Q * N) where Q = number of updates, N = array size

For 10,000 updates on array of 10,000:

  • ❌ Naive (worst case): 10,000 × 10,000 = 100,000,000 operations
  • ✅ Difference Array: 20,000 (mark starts/ends) + 10,000 (prefix sum) = 30,000 operations
  • 🚀 Speed up: ~3333x faster!

The Brilliant Trick:

Instead of updating every element in a range, we only mark two positions:

1 At range START: Add +value (signals "increment starts here")
2 At range END+1: Add -value (signals "increment stops here")

Think of it Like a Staircase:

Imagine walking through the array, tracking "current height":

  • When you hit +value, you step UP by that amount
  • When you hit -value, you step DOWN by that amount
  • Your current height is the accumulated change = prefix sum!

Visual Example:

Add 3 to range [1, 3] in array [0, 0, 0, 0, 0]

                   ↓

Step-by-step walk:
Index 0: height = 0
Index 1: height = 0 + 3 = 3 ⬆️ (stepped up)
Index 2: height = 3 + 0 = 3 ➡️ (plateau)
Index 3: height = 3 + 0 = 3 ➡️ (plateau)
Index 4: height = 3 - 3 = 0 ⬇️ (stepped down)
Index 5: height = 0 + 0 = 0 ➡️ (back to ground)

Why Prefix Sum is Perfect Here:

1 Compact representation: Instead of storing final values, we store just the changes
2 Efficient updates: Each range update = just 2 array modifications (O(1))
3 Batch processing: Apply ALL updates first, THEN compute final state once
4 Reconstruction: Prefix sum "unfolds" the difference array into actual values

The Mathematical Connection:

The difference array is the inverse of prefix sum:

prefix_sum(difference_array) = original_array
difference_array[i] = original[i] - original[i-1]

Real-World Applications:

  • ✈️ Flight bookings: Reserve seats for range of flights
  • 🚗 Carpooling: Track passengers in car over route segments
  • 📅 Calendar events: Mark busy times in schedule
  • 🎮 Game buffs: Apply temporary stat boosts over time ranges
  • 🏢 Resource allocation: Assign resources to time slots

When to Use This Pattern:

Use Difference Array when:

  • ✅ You have MANY range updates
  • ✅ Updates come all at once (batch processing)
  • ✅ You need final state AFTER all updates
  • ✅ Ranges can overlap

DON'T use when:

  • ❌ Need array state DURING updates (use Segment Tree instead)
  • ❌ Single update only (just update directly)
  • ❌ Need to query ranges while updating (use Fenwick Tree or Segment Tree)

The Key Insight:

Prefix sum and difference array are two sides of the same coin:

  • Prefix Sum: Expands differences into cumulative values (integration)
  • Difference Array: Compresses values into differences (differentiation)

Together, they give us O(1) range updates and O(N) final reconstruction!

Example: Add 3 to range [1, 3]

Difference array technique:

1 diff[1] += 3 (start of range)
2 diff[4] -= 3 (end+1 of range)
3 Apply prefix sum on diff to get final array 
public int[] rangeUpdate(int[] arr, int[][] updates) {
    int n = arr.length;
    int[] diff = new int[n + 1];
    
    // Apply all updates to difference array
    for (int[] update : updates) {
        int start = update[0];
        int end = update[1];
        int val = update[2];
        
        diff[start] += val;
        diff[end + 1] -= val;
    }
    
    // Build result using prefix sum
    int[] result = new int[n];
    result[0] = arr[0] + diff[0];
    
    for (int i = 1; i < n; i++) {
        diff[i] += diff[i - 1];
        result[i] = arr[i] + diff[i];
    }
    
    return result;
}

Pattern 7: Prefix XOR

Problem: XOR of subarray, find pairs with XOR = K

Key Insight: XOR has self-inverse property: A ⊕ A = 0

Time: O(N), Space: O(N)

🤔 Why Use Prefix XOR for This Pattern?

The XOR Property:

XOR has a unique mathematical property that makes it perfect for prefix operations:

A ⊕ A = 0 (anything XOR itself = 0)
A ⊕ 0 = A (XOR with 0 = identity)
A ⊕ B = B ⊕ A (commutative)

Key Insight for Subarrays:

If we have prefix XOR from 0 to j and from 0 to i-1:

XOR(i, j) = prefix[j] ⊕ prefix[i-1]

Why? Because:

1 prefix[j] = arr[0] ⊕ arr[1] ⊕ ... ⊕ arr[j]
2 prefix[i-1] = arr[0] ⊕ arr[1] ⊕ ... ⊕ arr[i-1]
3 prefix[j] ⊕ prefix[i-1] cancels out arr[0]...arr[i-1] (they XOR to 0!)
4 Left with: arr[i] ⊕ arr[i+1] ⊕ ... ⊕ arr[j] ✓

Finding Subarrays with XOR = K:

Similar to Pattern 2, but with XOR instead of sum:

If prefix[j] ⊕ prefix[i-1] = K, then:

prefix[i-1] = prefix[j] ⊕ K

So we look for (currentXOR ⊕ K) in our HashMap!

Real-World Applications:

  • 🔐 Cryptography (XOR encryption)
  • 🎮 Game state XOR (Nim game, sprague-grundy)
  • 💾 Error detection (parity bits)
  • 🧮 Finding unique elements
public int subarraysWithXOR(int[] arr, int k) {
    Map map = new HashMap<>();
    int count = 0;
    int xor = 0;
    
    map.put(0, 1);  // Important: XOR of empty subarray is 0
    
    for (int num : arr) {
        xor ^= num;  // Current prefix XOR
        
        // Check if (xor ^ k) exists - means we found subarray with XOR = k
        int target = xor ^ k;
        if (map.containsKey(target)) {
            count += map.get(target);
        }
        
        // Add current XOR to map
        map.put(xor, map.getOrDefault(xor, 0) + 1);
    }
    
    return count;
}

📊 Example: arr = [4, 2, 2, 6, 4], k = 6

Step-by-step XOR accumulation:

Index Value Prefix XOR Looking for Found?
0 4 4 4 ⊕ 6 = 2 ❌
1 2 6 6 ⊕ 6 = 0 ✅ count++
2 2 4 4 ⊕ 6 = 2 ❌
3 6 2 2 ⊕ 6 = 4 ✅ count += 2
4 4 6 6 ⊕ 6 = 0 ✅ count++

Total count = 4 subarrays with XOR = 6:

  1. [4, 2] → 4 ⊕ 2 = 6 ✓
  2. [2, 2, 6] → 2 ⊕ 2 ⊕ 6 = 6 ✓
  3. [6] → 6 ✓
  4. [2, 6, 4] → 2 ⊕ 6 ⊕ 4 = 6 ✓

Pattern 8: Prefix Sum with Modulo (Divisibility)

Problem: Find subarrays divisible by K or with sum % K = 0

Key Insight: Track prefix sum modulo K, use HashMap

Time: O(N), Space: O(K)

🤔 Why Use Prefix Sum with Modulo?

The Modulo Property:

If we want subarrays where sum is divisible by K:

(prefix[j] - prefix[i-1]) % K = 0

This means:

prefix[j] % K = prefix[i-1] % K

Key Insight:

If two positions have the SAME remainder when divided by K, the subarray between them is divisible by K!

Handling Negative Numbers:

In many languages, modulo of negative numbers can be negative. We need to fix this:

// Wrong: can give negative remainder
int mod = sum % k;

// Correct: always gives positive remainder
int mod = ((sum % k) + k) % k;

Why This Matters:

Example: sum = -7, k = 5

  • Wrong: -7 % 5 = -2 (in Java/C++)
  • Correct: ((-7 % 5) + 5) % 5 = (-2 + 5) % 5 = 3

Both -2 and 3 represent the same equivalence class modulo 5, but we need consistent non-negative values for HashMap!

Real-World Applications:

  • 📅 Finding weeks (sum divisible by 7)
  • 💰 Currency conversion (divisible by exchange rate)
  • 📦 Packaging problems (items divisible by box size)
  • ⏰ Time calculations (hours divisible by 24)
public int subarraysDivByK(int[] arr, int k) {
    Map map = new HashMap<>();
    int count = 0;
    int sum = 0;
    
    map.put(0, 1);  // Remainder 0 seen once initially
    
    for (int num : arr) {
        sum += num;
        
        // CRITICAL: Handle negative remainders
        int mod = ((sum % k) + k) % k;
        
        // If this remainder seen before, found valid subarrays
        if (map.containsKey(mod)) {
            count += map.get(mod);
        }
        
        map.put(mod, map.getOrDefault(mod, 0) + 1);
    }
    
    return count;
}

📊 Example: arr = [4, 5, 0, -2, -3, 1], k = 5

Find all subarrays with sum divisible by 5:

Index Value Sum Sum % 5 Count Update
- - 0 0 map[0] = 1
0 4 4 4 count += 0
1 5 9 4 count += 1 (found [5]!)
2 0 9 4 count += 2 (found 2 more!)
3 -2 7 2 count += 0
4 -3 4 4 count += 3 (found 3 more!)
5 1 5 0 count += 1 (entire array!)

Total = 7 subarrays divisible by 5

💡 Why Remainder 4 appeared 4 times?

At indices 0, 1, 2, and 4, the prefix sum % 5 = 4.

This means there are C(4,2) = 6 subarrays between these positions with sum divisible by 5!

Plus the subarray from start with remainder 0 = 1 more.

Total = 6 + 1 = 7 ✓

🚫 Common Mistakes

Mistake 1: Array Index Out of Bounds

Problem: Accessing prefix[left - 1] when left = 0

Solution: Always check if left == 0 before subtracting

// Wrong
return prefix[right] - prefix[left - 1];

// Correct
if (left == 0) return prefix[right];
return prefix[right] - prefix[left - 1];

Mistake 2: Forgetting map.put(0, 1)

Problem: Missing subarrays that start from index 0

Solution: Initialize HashMap with {0: 1} before loop

Mistake 3: Using Prefix Sum for Maximum Subarray

Problem: Trying to find max sum using prefix sum

Why Wrong: Prefix sum finds sum = K, not maximum sum

Correct Approach: Use Kadane's Algorithm for maximum sum

Mistake 4: Integer Overflow

Problem: Sum exceeds Integer.MAX_VALUE

Solution: Use long instead of int for prefix array

long[] prefix = new long[n];  // Not int[]

Mistake 5: Modifying Original Array

Problem: Computing prefix sum in-place destroys original data

Solution: Create separate prefix array or work backwards

🔄 Prefix Sum vs Similar Techniques

Technique When to Use Time Complexity Key Difference
Prefix Sum Sum of subarray = K, multiple range queries O(N) + O(1) query Cumulative from start
Sliding Window Fixed/variable window, min/max in window O(N) Maintains window state
Kadane's MAXIMUM sum subarray O(N) Tracks best so far
Two Pointers Sorted array, find pairs/triplets O(N) or O(N²) Moves from both ends
Segment Tree Range queries + updates O(log N) query/update Tree structure

🎓 Decision Tree: Which Technique?

1 Is it about subarray/range sum? → Consider Prefix Sum
2 Are there multiple queries? → Definitely Prefix Sum
3 Need exact sum = K? → Prefix Sum + HashMap
4 Need MAXIMUM sum? → Kadane's Algorithm (NOT prefix sum)
5 Fixed window size? → Sliding Window (NOT prefix sum)
6 Need to update values? → Difference Array or Segment Tree

🎯 Quick Reference Cheat Sheet

Core Formulas

prefix[i] = prefix[i-1] + arr[i]
sum(L, R) = prefix[R] - prefix[L-1]
If prefix[j] - prefix[i] = K, then i = j - K
2D: prefix[i][j] = top + left - diagonal + current

Pattern Recognition

Use Prefix Sum when you see:

  • "Sum of subarray/range"
  • "Multiple queries"
  • "Count subarrays with sum = K"
  • "Continuous/contiguous elements"

DON'T use Prefix Sum for:

  • Finding MAXIMUM sum (use Kadane's)
  • Fixed-size windows (use Sliding Window)
  • Sorted pair finding (use Two Pointers)
