⛰️ Priority Queue - The Ultimate Visual Guide

📚 Part 1: What is a Priority Queue?

A Priority Queue is a data structure where each element has a priority, and elements are served based on their priority, NOT the order they were added.
  • Highest priority element is always served first
  • Like a hospital ER: Critical patients (high priority) go before minor injuries (low priority)
  • NOT First-In-First-Out (FIFO) like a regular queue!

🎯 Key Insight: Priority Queue gives you the min (or max) element instantly!
Perfect for "find smallest k", "merge k lists", "top k frequent", etc.

How is it Implemented? THE HEAP!

Priority Queues are typically implemented using a Binary Heap:
  • Min-Heap: Parent ≤ Children (smallest at top)
  • Max-Heap: Parent ≥ Children (largest at top)
  • Stored in an array for efficiency!
  • NOT a Binary Search Tree (different structure!)

Priority Queue Operations

Operation Description Time Complexity
add(element) Insert element and maintain heap property O(log n)
poll() Remove and return min/max element O(log n)
peek() View min/max element without removing O(1)
size() Get number of elements O(1)
isEmpty() Check if empty O(1)

Java PriorityQueue - The Basics

// Import import java.util.PriorityQueue; // Create Min-Heap (default - smallest element first) PriorityQueue<Integer> minHeap = new PriorityQueue<>(); // Create Max-Heap (largest element first) PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder()); // OR using lambda: PriorityQueue<Integer> maxHeap = new PriorityQueue<>((a, b) -> b - a); // Basic Operations minHeap.add(5); // Add element - O(log n) minHeap.offer(3); // Same as add - O(log n) int min = minHeap.peek(); // View top element - O(1) int removed = minHeap.poll(); // Remove top element - O(log n) int size = minHeap.size(); // Get size - O(1) boolean empty = minHeap.isEmpty(); // Check if empty - O(1)

🔧 Part 2: How Heap Operations Work (Visual)

Operation 1: Add (Insert) Element

Goal: Add element while maintaining Min-Heap property (parent ≤ children)

📋 Algorithm Steps for Add (Bubble Up)

Step 1: Add the new element to the end of the array
    • This is index = size of heap
    • Temporarily violates heap property!

Step 2: Bubble Up - Compare with parent and swap if needed
    • Calculate parent index: parent = (current - 1) / 2
    • If new element < parent → SWAP (violates min-heap!)
    • If new element ≥ parent → STOP (heap property satisfied!)

Step 3: Repeat Step 2 with new position
    • Keep bubbling up until element ≥ parent OR reach root

Time Complexity: O(log n) - At most log n levels to bubble up

Operation 2: Poll (Remove Min)

Goal: Remove and return the minimum element (root) while maintaining Min-Heap property

📋 Algorithm Steps for Poll (Bubble Down)

Step 1: Save root element (index 0) - this is what we'll return
    • This is the minimum in min-heap

Step 2: Move last element to root position
    • Remove last element from array
    • Place it at index 0
    • Temporarily violates heap property!

Step 3: Bubble Down - Compare with children and swap if needed
    • Calculate child indices:
      - Left child: 2 * current + 1
      - Right child: 2 * current + 2
    • Find the smallest of current and both children
    • If current > smallest child → SWAP with smallest child
    • If current ≤ both children → STOP (heap property satisfied!)

Step 4: Repeat Step 3 with new position
    • Keep bubbling down until element ≤ both children OR reach leaf

Step 5: Return the saved root value

Time Complexity: O(log n) - At most log n levels to bubble down

Summary of Bubble Down Logic:

At each position:
1. Calculate left child (2*i+1) and right child (2*i+2)
2. Find smallest among: current, left child, right child
3. If current = smallest → STOP (heap property satisfied!) ✅
4. If smallest is a child → SWAP with that child and continue from child's position
5. If no children exist → STOP (reached leaf node) ✅
💡 Key Insight:
Add: O(log n) - bubble up at most log n levels
Poll: O(log n) - bubble down at most log n levels
Peek: O(1) - just look at index 0!

Max-Heap Operations: Add & Poll

📋 Max-Heap Add Algorithm (Bubble Up)

Only ONE thing changes from Min-Heap:

Step 1: Add to end (SAME as min-heap)

Step 2: Bubble Up with REVERSED comparison
    • Calculate parent: parent = (current - 1) / 2
    • If new element > parent → SWAP (violates max-heap!)
    • If new element ≤ parent → STOP (heap property satisfied!)

Step 3: Repeat until element ≤ parent OR reach root

The key difference: We swap when child is LARGER than parent (opposite of min-heap)

📋 Max-Heap Poll Algorithm (Bubble Down)

What changes from Min-Heap:

Step 1: Save root - Maximum element (SAME as min-heap concept)

Step 2: Move last to root (SAME)

Step 3: Bubble Down with REVERSED logic
    • Calculate children: left = 2*i+1, right = 2*i+2
    • Find LARGEST of current and both children (not smallest!)
    • If current < largest child → SWAP with largest child
    • If current ≥ both children → STOP

Step 4: Repeat until satisfied or reach leaf

The key difference: We find LARGEST child and swap if current is SMALLER (opposite!)

🎯 PATTERN 1: Top K Elements (K Smallest/Largest)

🔍 How to Recognize This Pattern?

You should think "Top K Elements" when you see:

✓ "K smallest elements"
✓ "K largest elements"
✓ "Kth smallest" or "Kth largest"
✓ "Top K frequent" elements
✓ "K closest points"
✓ Any problem with "K" and "smallest/largest/most/least"

Keywords to watch for:
"top k", "k smallest", "k largest", "kth element", "k closest", "k frequent"

💡 The Intuition - Why Priority Queue?

The Problem:
Find K smallest elements from [3, 1, 5, 12, 2, 11], K=3

Naive approach: Sort entire array → O(n log n)
Then take first k elements

🎯 Smart approach with Priority Queue:

For K SMALLEST: Use MAX-HEAP of size K!
• If element < heap top → Remove top, add element
• Heap always has K smallest!
• Time: O(n log k) - Better than O(n log n) when k << n

Why MAX-HEAP for K smallest?
Think of it as a "bouncer" keeping only K smallest:
• Top of heap = largest of our K smallest
• New element smaller? Kick out the largest!
• New element larger? Reject it!

Similarly, for K LARGEST: Use MIN-HEAP!
• Top = smallest of our K largest
• New element larger? Kick out smallest!

📋 Why This Pattern Applies

Problem: Find 3 smallest elements from [3, 1, 5, 12, 2, 11]
Why Max-Heap? Keep K elements, reject larger ones → top is boundary!
Alternative: Sort O(n log n) - Priority Queue is O(n log k)
When k is small: Much faster! (e.g., top 10 from 1 million)

// Pattern 1: K Smallest Elements using Max-Heap public static List<Integer> kSmallest(int[] nums, int k) { // Max-Heap (largest on top) PriorityQueue<Integer> maxHeap = new PriorityQueue<>((a, b) -> b - a); for (int num : nums) { maxHeap.add(num); // If heap size exceeds k, remove largest if (maxHeap.size() > k) { maxHeap.poll(); // Remove top (largest) } } return new ArrayList<>(maxHeap); } // Time: O(n log k) - Process n elements, heap operations log k // Space: O(k) - Heap size limited to k // Pattern 1: K Largest Elements using Min-Heap public static List<Integer> kLargest(int[] nums, int k) { // Min-Heap (smallest on top) PriorityQueue<Integer> minHeap = new PriorityQueue<>(); for (int num : nums) { minHeap.add(num); if (minHeap.size() > k) { minHeap.poll(); // Remove top (smallest) } } return new ArrayList<>(minHeap); } // Time: O(n log k), Space: O(k)

🎯 PATTERN 2: Merge K Sorted Lists/Arrays

🔍 How to Recognize This Pattern?

You should think "Merge K Sorted" when you see:

✓ "Merge K sorted lists"
✓ "Merge K sorted arrays"
✓ "Find smallest range" covering k lists
✓ Multiple sorted inputs to combine
✓ Need to maintain sorted order while merging

Keywords: "merge", "k lists", "sorted", "combine"

💡 The Intuition

The Problem:
Merge K=3 sorted lists:
L1: 1→4→5
L2: 1→3→4
L3: 2→6

Naive approach: Collect all, sort → O(N log N) where N = total elements

🎯 Smart approach with Min-Heap:

Key Insight: Next smallest element MUST be at the head of one of the k lists!

Algorithm:
1. Put all k heads in min-heap
2. Poll minimum (that's next in result!)
3. Add the next element from that list to heap
4. Repeat!

Why this works:
• At any point, heap has ≤ k elements (one from each list)
• Poll gives smallest among all current heads
• That's the global next smallest!

Time: O(N log k) where N = total elements
Better than O(N log N) when k << N!

Visual: Merge [1→4], [1→3], [2→6]

Step 1: Add all heads to min-heap
L1: 1→4
L2: 1→3
L3: 2→6
Min-Heap: [1(L1), 1(L2), 2(L3)]
Step 2: Poll min = 1 from L1, add next from L1 (4)
Result: [1]
Min-Heap: [1(L2), 2(L3), 4(L1)]
Step 3: Poll min = 1 from L2, add next from L2 (3)
Result: [1, 1]
Min-Heap: [2(L3), 3(L2), 4(L1)]
Step 4: Poll min = 2 from L3, add next from L3 (6)
Result: [1, 1, 2]
Min-Heap: [3(L2), 4(L1), 6(L3)]
Continue... Poll 3, 4, 4, 5, 6
Final Result: [1, 1, 2, 3, 4, 4, 5, 6] ✅
// Pattern 2: Merge K Sorted Lists public ListNode mergeKLists(ListNode[] lists) { if (lists == null || lists.length == 0) return null; // Min-Heap comparing by node values PriorityQueue<ListNode> minHeap = new PriorityQueue<>((a, b) -> a.val - b.val); // Add all list heads to heap for (ListNode head : lists) { if (head != null) { minHeap.add(head); } } ListNode dummy = new ListNode(0); ListNode curr = dummy; while (!minHeap.isEmpty()) { // Poll smallest ListNode smallest = minHeap.poll(); curr.next = smallest; curr = curr.next; // Add next from same list if (smallest.next != null) { minHeap.add(smallest.next); } } return dummy.next; } // Time: O(N log k) where N = total nodes, k = number of lists // Space: O(k) - Heap size

🎯 PATTERN 3: Two Heaps (Find Median from Data Stream)

🔍 How to Recognize This Pattern?

You should think "Two Heaps" when you see:

✓ "Find median" from data stream
✓ "Running median"
✓ Need to access middle element(s) efficiently
✓ "Sliding window median"
✓ Problems needing both min and max of different parts

Keywords: "median", "middle", "stream", "running"

💡 The Intuition - The Two Heaps Trick

The Challenge:
Find median of: 1, 2, 3, 4, 5
Median = 3 (middle element)

If we had sorted array: Just return middle → O(1)
But we're getting stream: Can't sort every time!

🎯 Brilliant Idea: Split in Half!

Keep two heaps:
• Max-Heap (left): Smaller half of numbers
• Min-Heap (right): Larger half of numbers

Example: [1, 2, 3, 4, 5]
Max-Heap (left): [1, 2, 3] → top is 3
Min-Heap (right): [4, 5] → top is 4

Median:
• If sizes equal → (maxHeap.peek() + minHeap.peek()) / 2
• If maxHeap larger → maxHeap.peek()

Balancing rule: |maxHeap.size() - minHeap.size()| ≤ 1

Where to add new element?
1. If ≤ maxHeap.top → add to maxHeap
2. Else → add to minHeap
3. Rebalance if needed!

Size: 2

Median = (5 + 7) / 2 = 6 ✅

// Pattern 3: Two Heaps - Find Median class MedianFinder { PriorityQueue<Integer> maxHeap; // Left half (smaller numbers) PriorityQueue<Integer> minHeap; // Right half (larger numbers) public MedianFinder() { maxHeap = new PriorityQueue<>((a, b) -> b - a); // Max heap minHeap = new PriorityQueue<>(); // Min heap (default) } public void addNum(int num) { // Add to appropriate heap if (maxHeap.isEmpty() || num <= maxHeap.peek()) { maxHeap.add(num); } else { minHeap.add(num); } // Rebalance to maintain size property if (maxHeap.size() > minHeap.size() + 1) { minHeap.add(maxHeap.poll()); } else if (minHeap.size() > maxHeap.size()) { maxHeap.add(minHeap.poll()); } } public double findMedian() { if (maxHeap.size() == minHeap.size()) { return (maxHeap.peek() + minHeap.peek()) / 2.0; } return maxHeap.peek(); } } // Time: addNum O(log n), findMedian O(1) // Space: O(n)

🎯 PATTERN 4: Scheduling & Meeting Rooms

🔍 How to Recognize This Pattern?

You should think "Scheduling with Heap" when you see:

✓ "Minimum meeting rooms" required
✓ "Task scheduler"
✓ "CPU scheduling"
✓ "Course schedule" with prerequisites
✓ Problems with overlapping intervals
✓ Need to track earliest ending task/meeting

Keywords: "meeting rooms", "schedule", "intervals", "overlapping", "minimum resources"

💡 The Intuition

Problem: Find minimum meeting rooms needed
Meetings: [[0,30], [5,10], [15,20]]

🎯 Think like a receptionist:

Approach:
1. Sort meetings by start time
2. Use min-heap to track end times of ongoing meetings
3. For each new meeting:
• If earliest ending meeting finishes before this starts → Reuse that room!
• Else → Need new room

Why min-heap?
• Need to know which meeting ends earliest
• That room becomes available first!
• Heap size = rooms currently in use
• Max heap size = answer!

Example:
Meeting [0,30] starts → Room 1, heap=[30]
Meeting [5,10] starts → 30>5, need Room 2, heap=[10,30]
Meeting [15,20] starts → 10<15, Room 2 free! heap=[20,30]
Max rooms used = 2 ✅

Visual: Meetings [[0,30], [5,10], [15,20]]

[0,30], [5,10], [15,20]
Process [0,30]: First meeting, need 1 room
Min-Heap (end times)
[30]

Rooms used: 1

Process [5,10]: Start=5, earliest end=30

5 < 30 → Meeting still ongoing, need NEW room!

Min-Heap (end times)

Rooms used: 2

15 ≥ 10 → First meeting ended, REUSE that room!

[20, 30]

Result:
Minimum Meeting Rooms = 2 ✅
// Pattern 4: Meeting Rooms II public int minMeetingRooms(int[][] intervals) { if (intervals.length == 0) return 0; // Sort by start time Arrays.sort(intervals, (a, b) -> a[0] - b[0]); // Min-heap of end times PriorityQueue<Integer> minHeap = new PriorityQueue<>(); for (int[] interval : intervals) { // If earliest ending meeting finishes before this starts if (!minHeap.isEmpty() && minHeap.peek() <= interval[0]) { minHeap.poll(); // Reuse that room } // Add current meeting's end time minHeap.add(interval[1]); } // Heap size = rooms needed return minHeap.size(); } // Time: O(n log n) - Sorting + heap operations // Space: O(n) - Heap can have all meetings

🎯 PATTERN 5: Sliding Window with Heap

🔍 How to Recognize This Pattern?

You should think "Sliding Window + Heap" when you see:

✓ "Sliding window maximum/minimum"
✓ "Moving average" of top k
✓ "Continuous median" in window
✓ Need max/min in current window
✓ Window slides and need to track extremes

Keywords: "sliding window", "window maximum", "window median", "stream"

💡 The Intuition

Problem: Find max in each window of size k
Array: [1,3,-1,-3,5,3,6,7], k=3

Naive: For each window, scan k elements → O(n×k)

🎯 With Max-Heap:
• Keep elements with their indices in heap
• Peek gives max, but check if it's in current window!
• Remove old elements (outside window)

Key insight: Lazy deletion!
Don't immediately remove elements leaving window
Just check at peek time: is top index still in window?
If not, poll it and check next

Time: O(n log n) - Better than O(n×k) when k is large!

• Window size k=3

🔑 Key Insight - Lazy Deletion:

We don't remove elements immediately when they leave the window
Instead, we check at peek time: "Is top element's index still in current window?"
• If YES → Use it as max
• If NO → Poll it and check next

This is more efficient than trying to remove specific elements from heap!
// Pattern 5: Sliding Window Maximum public int[] maxSlidingWindow(int[] nums, int k) { int[] result = new int[nums.length - k + 1]; // Max-heap storing [value, index] PriorityQueue<int[]> maxHeap = new PriorityQueue<>((a, b) -> b[0] - a[0]); for (int i = 0; i < nums.length; i++) { maxHeap.add(new int[]{nums[i], i}); // Remove elements outside current window while (maxHeap.peek()[1] <= i - k) { maxHeap.poll(); } // Record max for this window if (i >= k - 1) { result[i - k + 1] = maxHeap.peek()[0]; } } return result; } // Time: O(n log n), Space: O(n) // Note: Deque solution is O(n) but heap is more intuitive!

⚠️ Common Pitfalls & How to Avoid Them

❌ Mistake #1: Wrong Heap Type for Top K

Wrong: Use min-heap for k smallest
Why wrong: Min-heap will keep ALL small elements, can't limit to k!

✅ Correct: Use MAX-heap for k smallest
Why: Top is the "bouncer" - largest of our k smallest
If new element smaller, kick out the largest!

❌ Mistake #2: Forgetting Custom Comparator

Problem: Sort by custom criteria (e.g., frequency, distance)

Wrong: PriorityQueue<int[]> pq = new PriorityQueue<>();
Why wrong: Arrays don't have natural ordering!

✅ Correct:
PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> a[0] - b[0]);

❌ Mistake #3: Not Maintaining Two Heaps Balance

For median problems:

Must ensure: |maxHeap.size() - minHeap.size()| ≤ 1

✅ Always rebalance after adding:
if (maxHeap.size() > minHeap.size() + 1) {
    minHeap.add(maxHeap.poll());
}

❌ Mistake #4: Integer Overflow in Comparator

Wrong: (a, b) -> a - b for large integers
Why wrong: a - b can overflow!

✅ Correct:
(a, b) -> Integer.compare(a, b)
OR
(a, b) -> a > b ? 1 : (a < b ? -1 : 0)
