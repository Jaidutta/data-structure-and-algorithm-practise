1. What is Recursion? 🤔

Simple Definition:

Recursion is when a function calls itself to solve a smaller version of the same problem. It breaks down a big problem into smaller, identical sub-problems.

Real-World Analogy: Russian Dolls 🪆

Opening Nested Dolls:

Open a doll:
  ├─ If empty → STOP (base case)
  └─ If has another doll inside:
       └─ Open that doll (recursive call)
            └─ Keep opening until you find the smallest one

Code Example: Simple Countdown

// Print numbers from n down to 1
void countdown(int n) {
    // BASE CASE: Stop condition
    if (n == 0) {
        System.out.println("Done!");
        return;
    }
    
    // Do something
    System.out.println(n);
    
    // RECURSIVE CASE: Call itself with smaller input
    countdown(n - 1);
}

// countdown(3) prints: 3, 2, 1, Done!

2. How Recursion Works (The Call Stack) 📚

Key Concept: Each function call is placed on a STACK. When a function calls itself, the new call is added on top. When a function returns, it's removed from the stack.

Visual Example: factorial(3)

int factorial(int n) {
    if (n == 1) return 1;  // base case
    return n * factorial(n - 1);  // recursive case
}
📚 CALL STACK EVOLUTION: Step 1: Call factorial(3) ┌─────────────────────┐ │ factorial(3) │ ← Top of stack (active) │ n=3, return 3*? │ Waiting for factorial(2) └─────────────────────┘ Step 2: factorial(3) calls factorial(2) ┌─────────────────────┐ │ factorial(2) │ ← Top (active) │ n=2, return 2*? │ Waiting for factorial(1) ├─────────────────────┤ │ factorial(3) │ PAUSED │ n=3, return 3*? │ └─────────────────────┘ Step 3: factorial(2) calls factorial(1) ┌─────────────────────┐ │ factorial(1) │ ← Top (active) │ n=1, return 1 │ BASE CASE! Returns immediately ├─────────────────────┤ │ factorial(2) │ PAUSED │ n=2, return 2*? │ ├─────────────────────┤ │ factorial(3) │ PAUSED │ n=3, return 3*? │ └─────────────────────┘ Step 4: factorial(1) returns 1 ⬆️ Returns 1 ┌─────────────────────┐ │ factorial(2) │ ← Top (active now) │ n=2, return 2*1=2 │ Calculates 2*1=2 ├─────────────────────┤ │ factorial(3) │ PAUSED │ n=3, return 3*? │ └─────────────────────┘ Step 5: factorial(2) returns 2 ⬆️ Returns 2 ┌─────────────────────┐ │ factorial(3) │ ← Top (active now) │ n=3, return 3*2=6 │ Calculates 3*2=6 └─────────────────────┘ Step 6: factorial(3) returns 6 ⬆️ Returns 6 └─────────────────────┘ Stack is empty! FINAL ANSWER: 6
💡 Key Insight: The stack grows (function calls) until it hits the base case, then shrinks (functions return) as results bubble back up.

3. Base Case & Recursive Case 🎯

Every Recursion Has 2 Parts:

  1. BASE CASE: The stopping condition (when to stop recursing)
  2. RECURSIVE CASE: The function calling itself with a smaller input

Template Structure:

ReturnType recursiveFunction(input) {
    // ========== PART 1: BASE CASE ==========
    // Check if we should STOP recursing
    if (/* stopping condition */) {
        return /* simple answer */;
    }
    
    // ========== PART 2: RECURSIVE CASE ==========
    // Break problem into smaller pieces
    // Call function with SMALLER input
    return /* combine */ recursiveFunction(/* smaller input */);
}

Examples of Base Cases:

Problem Type Base Case Why?
Factorial(n) if (n == 1) return 1 1! = 1 (simplest case)
Fibonacci(n) if (n <= 1) return n fib(0)=0, fib(1)=1 (known values)
Tree traversal if (node == null) return Empty tree - nothing to process
Array sum if (index >= arr.length) return 0 Reached end of array
String reverse if (str.length() == 0) return "" Empty string can't be reversed
⚠️ Common Mistake: Forgetting the base case leads to INFINITE RECURSION → Stack Overflow Error!

Pattern 1: Linear Recursion 📏

What it is: Function makes ONE recursive call per execution
When to use: Processing arrays, lists, strings sequentially
Key trait: Goes through elements one by one

Visualization: Sum of Array

sum([1, 2, 3, 4])
  ↓
1 + sum([2, 3, 4])
      ↓
    2 + sum([3, 4])
          ↓
        3 + sum([4])
              ↓
            4 + sum([])
                  ↓
                return 0 (base case)
                  ⬆️ 0
              ⬆️ 4
          ⬆️ 7
      ⬆️ 9
  ⬆️ 10

Code Example: Sum of Array

/**
 * LINEAR RECURSION: One recursive call per execution
 * Pattern: Process current + recurse on rest
 */
int sum(int[] arr, int index) {
    // BASE CASE: Reached end of array
    if (index >= arr.length) return 0;
    
    // RECURSIVE CASE: current element + sum of rest
    return arr[index] + sum(arr, index + 1);
}

// Time: O(n) - visits each element once
// Space: O(n) - call stack depth = array length

Common Linear Recursion Problems:

  • ✓ Sum/product of array elements
  • ✓ Find max/min in array
  • ✓ Reverse a string
  • ✓ Check if string is palindrome
  • ✓ Count occurrences in array

Pattern 2: Binary Recursion 🌿

What it is: Function makes TWO recursive calls per execution
When to use: Problems where solution depends on TWO sub-solutions
Key trait: Splits into two branches like a tree

Visualization: Fibonacci(4)

                      fib(4)
                    ╱           ╲
                  ╱               ╲
            fib(3)           fib(2)
            ╱     ╲             ╱     ╲
        fib(2) fib(1)     fib(1) fib(0)
        ╱   ╲       ↓           ↓       ↓
    fib(1) fib(0)  1       1   0
       ↓       ↓
      1   0

Returns bubble up: 1+0=1, 1+1=2, 2+1=3
Final Answer: 3

Code Example: Fibonacci

/**
 * BINARY RECURSION: Two recursive calls per execution
 * Pattern: Combine results from two sub-problems
 */
int fibonacci(int n) {
    // BASE CASES: First two numbers are known
    if (n <= 1) return n;
    
    // RECURSIVE CASE: Sum of previous two numbers
    return fibonacci(n - 1) + fibonacci(n - 2);
}

// Time: O(2^n) - exponential! (without memoization)
// Space: O(n) - maximum depth of recursion tree
⚠️ Performance Warning: Binary recursion can be VERY slow for large inputs (exponential time). Use memoization to optimize!

Common Binary Recursion Problems:

  • ✓ Fibonacci sequence
  • ✓ Climbing stairs (n ways)
  • ✓ Count paths in grid
  • ✓ Tower of Hanoi
  • ✓ Binary tree height

Pattern 3: Tree Recursion 🌳

What it is: Recursion on tree data structures (binary trees, N-ary trees)
When to use: Any binary tree problem (traversal, search, modification)
Key trait: Recurse on left child, then right child (or all children)

Visualization: Tree Height

Tree:          1
             ╱   ╲
           2       3
         ╱  ╲
       4     5

maxDepth(1):
  ├─ maxDepth(2):
  │    ├─ maxDepth(4): return 1 (leaf)
  │    └─ maxDepth(5): return 1 (leaf)
  │    return 1 + max(1,1) = 2
  └─ maxDepth(3): return 1 (leaf)
  return 1 + max(2,1) = 3

Code Example: Tree Height

/**
 * TREE RECURSION: Recurse on tree structure
 * Pattern: Base case (null), process left, process right
 */
int maxDepth(TreeNode root) {
    // BASE CASE: Empty tree has height 0
    if (root == null) return 0;
    
    // RECURSIVE CASE: Get heights of subtrees
    int leftHeight = maxDepth(root.left);
    int rightHeight = maxDepth(root.right);
    
    // Current height = 1 + max of subtrees
    return 1 + Math.max(leftHeight, rightHeight);
}

// Time: O(n) - visit each node once
// Space: O(h) - h = height of tree (call stack)

Tree Recursion Patterns:

Pattern When to Use Example
Preorder Process node before children Copy tree, serialize
Inorder Process node between children (BST) Get sorted values
Postorder Process node after children Delete tree, calculate height
Return Value Need result from children first Height, diameter, sum

Pattern 4: Backtracking 🔙

What it is: Try all possibilities, backtrack when invalid
When to use: Generate all combinations, permutations, subsets
Key trait: Make choice → recurse → undo choice (backtrack)

The Backtracking Template:

/**
 * BACKTRACKING PATTERN
 * 1. Choose - make a decision
 * 2. Explore - recurse with that choice
 * 3. Unchoose - undo the choice (backtrack)
 */
void backtrack(state, choices) {
    // BASE CASE: Found complete solution
    if (/* is solution complete */) {
        result.add(/* copy of current state */);
        return;
    }
    
    // TRY each choice
    for (choice : choices) {
        // 1. CHOOSE: Make the choice
        state.add(choice);
        
        // 2. EXPLORE: Recurse
        backtrack(state, remainingChoices);
        
        // 3. UNCHOOSE: Backtrack (undo)
        state.remove(choice);
    }
}

Example: Generate All Subsets

// Generate all subsets of [1,2,3]
void subsets(int[] nums, int start, List<Integer> current) {
    // Add current subset to result
    result.add(new ArrayList<>(current));
    
    for (int i = start; i < nums.length; i++) {
        current.add(nums[i]);         // Choose
        subsets(nums, i + 1, current);  // Explore
        current.remove(current.size() - 1);  // Unchoose
    }
}

Common Backtracking Problems:

  • ✓ Generate all permutations
  • ✓ Generate all combinations
  • ✓ Generate all subsets
  • ✓ N-Queens problem
  • ✓ Sudoku solver
  • ✓ Word search in grid
  • ✓ Palindrome partitioning

Pattern 5: Divide & Conquer ⚔️

What it is: Split problem in half, solve each half, combine
When to use: Sorting, searching in sorted arrays
Key trait: Divide → Conquer (recurse) → Combine results

The 3-Step Pattern:

1. DIVIDE: Split problem into smaller sub-problems
2. CONQUER: Solve sub-problems recursively
3. COMBINE: Merge solutions of sub-problems

Example: Binary Search

/**
 * DIVIDE & CONQUER: Binary Search
 * Divide: Split array in half
 * Conquer: Search in one half
 * Combine: Return result (no merge needed)
 */
int binarySearch(int[] arr, int target, int left, int right) {
    // BASE CASE: Not found
    if (left > right) return -1;
    
    // DIVIDE: Find middle
    int mid = left + (right - left) / 2;
    
    // Found it!
    if (arr[mid] == target) return mid;
    
    // CONQUER: Search in one half
    if (arr[mid] > target)
        return binarySearch(arr, target, left, mid - 1);
    else
        return binarySearch(arr, target, mid + 1, right);
}

// Time: O(log n) - halves search space each time

Pattern 6: Memoization (Top-Down DP) 💾

What it is: Cache results of expensive recursive calls
When to use: Overlapping sub-problems (same calculation repeated)
Key trait: Check cache before recursing, store result after

Pattern: Cache + Recursion

/**
 * MEMOIZATION PATTERN
 * Before: O(2^n) exponential
 * After: O(n) with caching!
 */
Map<Integer, Integer> memo = new HashMap<>();

int fib(int n) {
    // CHECK CACHE: Already computed?
    if (memo.containsKey(n)) return memo.get(n);
    
    // BASE CASE
    if (n <= 1) return n;
    
    // COMPUTE & CACHE result
    int result = fib(n - 1) + fib(n - 2);
    memo.put(n, result);
    
    return result;
}

// Without memo: O(2^n) - exponential
// With memo: O(n) - each fib(i) computed once!

6. Building Recursion Intuition 🧠

The 3-Step Thinking Process:

  1. Identify the BASE CASE: What's the simplest input?
  2. Trust the recursion: Assume it works for smaller input
  3. Make it work for current input: Use result from smaller input

Example: Factorial Thinking Process

Problem: Calculate n! = n × (n-1) × ... × 2 × 1

Step 1: What's the base case?

→ Simplest: 1! = 1 (or 0! = 1)

Step 2: Trust recursion for smaller input

→ Assume factorial(n-1) correctly returns (n-1)!

Step 3: Use that to solve current problem

→ n! = n × (n-1)!

→ So: return n * factorial(n-1)

The "Leap of Faith" 🪂

Key Insight: Don't try to trace the entire recursion in your head!

Instead:

  1. Assume the recursive call WORKS for smaller input
  2. Figure out how to use that result for current input
  3. Write the base case to stop recursion

That's it! Let the computer handle the details.

Common Mental Blocks & Solutions:

Mental Block Solution
"I can't visualize all the calls!" Don't try! Just focus on: base case + one recursive step
"How does it know to stop?" The base case stops it. Make sure input gets SMALLER each call
"It seems like magic..." It's just the call stack! Each call waits for the next to finish
"When should I use recursion?" When problem can be defined in terms of smaller version of itself

8. Common Mistakes & Pro Tips 💡

❌ Common Mistakes to Avoid:

1. Forgetting the Base Case

Problem: Infinite recursion → Stack Overflow

Solution: ALWAYS write base case first!

// ❌ BAD: No base case
int sum(arr, i) {
    return arr[i] + sum(arr, i + 1);  // Never stops!
}

// ✅ GOOD: Has base case
int sum(arr, i) {
    if (i >= arr.length) return 0;  // Stop condition
    return arr[i] + sum(arr, i + 1);
}

2. Not Making Input Smaller

Problem: Infinite recursion if input doesn't decrease

Solution: Each recursive call must move toward base case

// ❌ BAD: Input doesn't decrease
int factorial(n) {
    if (n == 1) return 1;
    return n * factorial(n);  // n stays same!
}

// ✅ GOOD: Input decreases
int factorial(n) {
    if (n == 1) return 1;
    return n * factorial(n - 1);  // n-1 is smaller
}

3. Modifying Backtracking State Without Undoing

Problem: Corrupted state affects other branches

Solution: Always undo changes (backtrack)

// ❌ BAD: No backtracking
void backtrack(list) {
    list.add(choice);
    backtrack(list);
    // Missing: list.remove(choice)
}

// ✅ GOOD: Properly backtrack
void backtrack(list) {
    list.add(choice);      // Choose
    backtrack(list);       // Explore
    list.remove(choice);   // Unchoose (backtrack)
}

4. Not Handling Edge Cases

Problem: Crashes on null, empty arrays, negative numbers

Solution: Check edge cases in base case

// ❌ BAD: Doesn't handle null
int maxDepth(TreeNode root) {
    return 1 + Math.max(maxDepth(root.left), maxDepth(root.right));
    // NullPointerException if root is null!
}

// ✅ GOOD: Handles null
int maxDepth(TreeNode root) {
    if (root == null) return 0;  // Handle edge case
    return 1 + Math.max(maxDepth(root.left), maxDepth(root.right));
}

✅ Pro Tips:

1. Start with Base Case

Always write the base case FIRST. It's easier to think about when to stop than how to recurse.

2. Test with Smallest Input

Test your function with the simplest possible input (empty array, n=0, null tree). If it works for that, it likely works for larger inputs.

3. Trust the Recursion (Leap of Faith)

Don't trace every recursive call in your head. Just make sure:

  • ✓ Base case is correct
  • ✓ Recursive call uses smaller input
  • ✓ You correctly use the recursive result

4. Draw the Recursion Tree (For Understanding)

When learning, draw the recursion tree for small inputs (like fib(4)). This helps you see the pattern, but don't do this for every problem!

5. Consider Memoization for Overlapping Subproblems

If you see the same calculation happening multiple times (like in Fibonacci), add a cache! This can turn O(2^n) into O(n).

6. Space Complexity = Max Depth of Recursion

Remember that recursion uses stack space. If you recurse n levels deep, that's O(n) space complexity from the call stack.
