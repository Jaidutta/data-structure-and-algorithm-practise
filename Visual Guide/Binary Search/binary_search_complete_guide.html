🔍 Binary Search - Complete Master Guide

1️⃣ What is Binary Search?

Binary Search is a search algorithm that finds the position of a target value within a sorted array. It works by repeatedly dividing the search interval in half. If the target is less than the middle element, search the left half; otherwise search the right half.
💡 Key Requirements: Array MUST be sorted!
Time Complexity: O(log n) - Much faster than linear search O(n)
Why? We eliminate HALF the array at each step!

2️⃣ How to Recognize Binary Search Problems

🎯 Recognition Patterns

Array/list is sorted or can be sorted
Searching for a specific value or position
Finding first/last occurrence
Problem involves "minimum/maximum value that satisfies condition"
Search space can be divided in half repeatedly
Keywords: "sorted", "find", "search", "minimum satisfying", "peak"
Need O(log n) time complexity
Problem about rotated sorted array
⚠️ Binary Search Won't Work When:
  • Array is unsorted AND cannot be sorted
  • Need to find ALL occurrences (but can find first/last)
  • Search space cannot be divided logically
  • Problem requires checking every element anyway

3️⃣ Three Binary Search Templates

There are three main templates for binary search, each suited for different scenarios. Understanding when to use each is crucial!

Template 1: Basic Binary Search

When to use: Finding exact value, search space reduced to 1 element
Termination: left > right
No need to check: After loop (found during loop or doesn't exist)
// Template 1: Basic Binary Search public static int binarySearch(int[] arr, int target) { int left = 0; int right = arr.length - 1; while (left <= right) { // Note: <= int mid = left + (right - left) / 2; // Avoid overflow if (arr[mid] == target) { return mid; // Found! } else if (arr[mid] < target) { left = mid + 1; // Search right half } else { right = mid - 1; // Search left half } } return -1; // Not found }

Template 2: Finding Boundaries

When to use: Find first/last occurrence, leftmost/rightmost position
Termination: left >= right (1 element left)
Post-processing: Check left or right after loop
// Template 2: Find First/Last Occurrence public static int findFirst(int[] arr, int target) { int left = 0; int right = arr.length - 1; while (left < right) { // Note: < (not <=) int mid = left + (right - left) / 2; if (arr[mid] < target) { left = mid + 1; // Target is on right } else { right = mid; // Target could be at mid or left } } // Post-processing: check if we found target return (arr[left] == target) ? left : -1; } public static int findLast(int[] arr, int target) { int left = 0; int right = arr.length - 1; while (left < right) { int mid = left + (right - left + 1) / 2; // Round up! if (arr[mid] > target) { right = mid - 1; // Target is on left } else { left = mid; // Target could be at mid or right } } return (arr[left] == target) ? left : -1; }

Template 3: Finding with Neighbors

When to use: Need to check element AND its neighbors (peak, valley)
Termination: left + 1 >= right (2 elements left)
Post-processing: Check both left and right
// Template 3: Search with Neighbor Checking public static int findPeak(int[] arr) { int left = 0; int right = arr.length - 1; while (left + 1 < right) { // Stop when 2 elements left int mid = left + (right - left) / 2; // Compare with neighbors if (arr[mid] < arr[mid + 1]) { left = mid; // Peak is on right } else { right = mid; // Peak is on left or at mid } } // Post-processing: check remaining 2 elements return arr[left] > arr[right] ? left : right; }
💡 Template Choice Summary:
Template 1: Exact match → Use left <= right
Template 2: First/Last position → Use left < right
Template 3: Need neighbors → Use left + 1 < right

public static int searchRotated(int[] arr, int target) { int left = 0, right = arr.length - 1; while (left <= right) { int mid = left + (right - left) / 2; if (arr[mid] == target) { return mid; } // Determine which half is sorted if (arr[left] <= arr[mid]) { // Left half is sorted if (target >= arr[left] && target < arr[mid]) { right = mid - 1; // Target in sorted left half } else { left = mid + 1; // Target in right half } } else { // Right half is sorted if (target > arr[mid] && target <= arr[right]) { left = mid + 1; // Target in sorted right half } else { right = mid - 1; // Target in left half } } } return -1; }

Variation 5: Search in Rotated Array II (With Duplicates)

Problem: Rotated array WITH duplicates [2,5,6,0,0,1,2]
Challenge: Can't determine which half is sorted when arr[left] == arr[mid]

💡 The Intuition

What's different with duplicates?

Without duplicates: [4,5,6,7,0,1,2]
• If arr[left] ≤ arr[mid] → left half is sorted ✓

With duplicates: [2,5,6,0,0,1,2]
• If arr[left] == arr[mid] == arr[right] = 2 → Can't tell which half is sorted! ❓

The Solution:
When stuck (all three equal), we can only shrink by 1 from both ends:
left++, right-- → Worst case becomes O(n) 😔

But when we CAN determine which half is sorted, we still eliminate half! 🎯

🤔 Why Binary Search Still Helps?

  • ✅ Best case: No duplicates at pivot points → O(log n)
  • ⚠️ Worst case: All elements same → O(n)
  • ✅ Average case: Usually better than O(n)
  • ✅ When we can determine sorted half, eliminate half the space
  • ✅ Only when ambiguous do we shrink by 1

Variation 6: Find Peak Element

Problem: Find any peak (element greater than neighbors)
Example: [1,2,3,1] → peak at index 2 (value 3)

💡 The Intuition

Why can we use Binary Search on unsorted array?

🎯 Key Insight: If we're at any element and one neighbor is greater, moving towards that neighbor GUARANTEES we'll find a peak!

Why?
• If arr[mid] < arr[mid+1] → Going right is "uphill"
• Either we hit a peak, OR we hit the end (which is also a peak by definition)
• We can NEVER get stuck without finding a peak!

The Magic: Even though array isn't sorted, we can still eliminate half the search space by following the "uphill" direction!

🤔 Why Binary Search?

  • ✅ Array doesn't need to be sorted!
  • ✅ We can eliminate half: Compare mid with mid+1
  • ✅ If arr[mid] < arr[mid+1] → Peak MUST be on right (going uphill)
  • ✅ If arr[mid] > arr[mid+1] → Peak could be mid or on left (going downhill)
  • ✅ Time: O(log n) instead of O(n) linear scan

public static int findPeakElement(int[] arr) { int left = 0, right = arr.length - 1; while (left < right) { int mid = left + (right - left) / 2; if (arr[mid] > arr[mid + 1]) { // Going downhill - peak is at mid or left right = mid; } else { // Going uphill - peak is on right left = mid + 1; } } return left; // left == right, peak found } // Time: O(log n), Space: O(1)

Variation 7: Find Minimum in Rotated Array

Problem: Find minimum in rotated sorted array
Example: [4,5,6,7,0,1,2] → return 0

💡 The Intuition

Understanding Rotation:

Original sorted: [0, 1, 2, 4, 5, 6, 7]
After rotation: [4, 5, 6, 7, 0, 1, 2]

🎯 Key Observation:
The array is split into TWO sorted parts:
• [4, 5, 6, 7] ← larger values
• [0, 1, 2] ← smaller values (contains minimum!)

The Pattern:
If arr[mid] > arr[right] → We're in the larger part, minimum is on RIGHT
If arr[mid] ≤ arr[right] → We're in smaller part or minimum is at mid or LEFT

We always compare mid with right, not left!

🤔 Why Binary Search?

  • ✅ Minimum is at the rotation pivot point
  • ✅ Can determine which half contains minimum by comparing mid with right
  • ✅ If arr[mid] > arr[right] → minimum is in right half
  • ✅ Otherwise → minimum is at mid or in left half
  • ✅ Time: O(log n) to find the pivot

Variation 8: Search in 2D Matrix

Problem: Search in m×n matrix where each row is sorted and first element of each row > last of previous
Example: Search for 3 in [[1,3,5,7],[10,11,16,20],[23,30,34,60]]

💡 The Intuition

The "Aha!" Moment:

Matrix: 
[1,  3,  5,  7]
[10, 11, 16, 20]
[23, 30, 34, 60]
🎯 Key Observation: This is really just a sorted array wrapped into a matrix!

If we "unwrap" it: [1, 3, 5, 7, 10, 11, 16, 20, 23, 30, 34, 60]

The Trick: We can treat it as 1D array of length m×n!
• Index 0 → row 0, col 0 → value 1
• Index 5 → row 1, col 1 → value 11
• Index 11 → row 2, col 3 → value 60

Formula:
• 1D index → row = index / n, col = index % n
• 2D (row, col) → index = row × n + col

🤔 Why Binary Search?

  • ✅ Matrix is fully sorted (treat as 1D sorted array)
  • ✅ Can convert between 1D index and 2D coordinates
  • ✅ Apply standard binary search on virtual 1D array
  • ✅ Time: O(log(m×n)) instead of O(m×n) linear search

Variation 9: Search in 2D Matrix II (Different Pattern)

Problem: Matrix where rows AND columns are sorted, but rows don't have the special property
Example: Search for 5 in [[1,4,7,11],[2,5,8,12],[3,6,9,16]]

💡 The Intuition

What's different from Variation 8?

Matrix: 
[1,  4,  7, 11]
[2,  5,  8, 12]
[3,  6,  9, 16]
Notice: 2 < 4 → Rows don't strictly follow each other!
We CAN'T treat it as a 1D sorted array 😔

🎯 The "Aha!" Moment:
Start at top-right corner (or bottom-left)!

At top-right (value=11):
• All numbers to the LEFT are smaller
• All numbers BELOW are larger

This gives us a binary decision!
• If target < current → Move LEFT (eliminate column)
• If target > current → Move DOWN (eliminate row)
• If target == current → Found! ✅

🤔 Why This Works (Not Pure BS)?

  • ✅ Each step eliminates either a full row OR full column
  • ✅ Starting at top-right gives us TWO sorted directions
  • ✅ Can't miss target - we systematically eliminate spaces
  • ⚠️ Time: O(m+n) not O(log n), but still better than O(m×n)!
  • 💡 Uses elimination principle like binary search

Variation 10: Square Root (Integer)

Problem: Find integer square root (round down)
Example: sqrt(8) → 2 (since 2² = 4 ≤ 8 < 9 = 3²)

💡 The Intuition

How do we think about this problem?

🎯 Reframe the question:
"What is sqrt(8)?" becomes "What is the largest integer k where k² ≤ 8?"

Search space: Numbers from 1 to 8
1² = 1 ✓ too small
2² = 4 ✓ still works
3² = 9 ✗ too big!

Answer: 2 (largest k where k² ≤ 8)

The Pattern: We're searching for a boundary in a monotonic space!
[1 ✓, 2 ✓, 3 ✗, 4 ✗, 5 ✗, ...] → Find last ✓

🤔 Why Binary Search?

  • ✅ Search space: 1 to x (or 1 to x/2 for optimization)
  • ✅ Property is monotonic: if k² > x, then (k+1)² > x too
  • ✅ Finding boundary: largest k where k² ≤ x
  • ✅ Can eliminate half: if mid² < x, answer is mid or larger
  • ✅ Time: O(log n) vs trying every number O(n)

Variation 11: Find Closest Element

Problem: Find element closest to target in sorted array
Example: [1,3,5,7,9], target = 6 → return 5 or 7 (both distance 1)

💡 The Intuition

How do we think about "closest"?

Array: [1, 3, 5, 7, 9], Target: 6

🎯 Two-step approach:
Step 1: Find where 6 would be inserted → between 5 and 7
Step 2: Compare the two neighbors!

Distance from 5: |5-6| = 1
Distance from 7: |7-6| = 1
Both equal! Return either (or left by convention)

The Pattern:
Use binary search to find insertion point,
then compare neighbors around that point!

🤔 Why Binary Search?

  • ✅ Array is sorted
  • ✅ Closest element must be near the insertion point
  • ✅ Binary search finds insertion point in O(log n)
  • ✅ Then O(1) comparison of neighbors
  • ✅ Total: O(log n)

public static int findClosest(int[] arr, int target) { int left = 0, right = arr.length - 1; // Find position where target would be inserted while (left < right) { int mid = left + (right - left) / 2; if (arr[mid] < target) { left = mid + 1; } else { right = mid; } } // Handle edge cases if (left == 0) return arr[0]; if (left == arr.length) return arr[arr.length - 1]; // Compare neighbors around insertion point int leftDiff = Math.abs(arr[left - 1] - target); int rightDiff = Math.abs(arr[left] - target); return leftDiff <= rightDiff ? arr[left - 1] : arr[left]; } // Time: O(log n), Space: O(1)

Variation 12: Find K Closest Elements

Problem: Find k elements closest to target
Example: [1,2,3,4,5], target = 3, k = 4 → [1,2,3,4]
public static List<Integer> findKClosest(int[] arr, int k, int target) { int left = 0, right = arr.length - k; // Binary search for the leftmost index of k elements while (left < right) { int mid = left + (right - left) / 2; // Compare distances: arr[mid] vs arr[mid+k] if (target - arr[mid] > arr[mid + k] - target) { left = mid + 1; // Right side is closer } else { right = mid; // Left side is closer or equal } } // Return k elements starting from left List<Integer> result = new ArrayList<>(); for (int i = left; i < left + k; i++) { result.add(arr[i]); } return result; }

5️⃣ Binary Search on Answer (Advanced Pattern)

Sometimes we binary search on the answer space rather than array indices. This is used when:
• Answer lies in a range [min, max]
• We can verify if a value works in O(n) or better
• If k works, then all values > k (or < k) also work

Template for Binary Search on Answer

// Binary Search on Answer Template public static int binarySearchOnAnswer() { int left = MIN_POSSIBLE_ANSWER; int right = MAX_POSSIBLE_ANSWER; while (left < right) { int mid = left + (right - left) / 2; if (isPossible(mid)) { right = mid; // Try smaller value } else { left = mid + 1; // Need larger value } } return left; } private static boolean isPossible(int value) { // Check if this value satisfies the condition // Usually O(n) check }

Example: Koko Eating Bananas (LeetCode 875)

Problem: Koko has piles of bananas. She can eat k bananas/hour. Find minimum k to finish all piles in h hours.
Insight: Binary search on k (eating speed), not on array!
public static int minEatingSpeed(int[] piles, int h) { int left = 1; // Minimum possible speed int right = 1_000_000_000; // Maximum possible speed while (left < right) { int mid = left + (right - left) / 2; if (canFinish(piles, mid, h)) { right = mid; // Can finish, try slower speed } else { left = mid + 1; // Too slow, need faster } } return left; } private static boolean canFinish(int[] piles, int k, int h) { int hoursNeeded = 0; for (int pile : piles) { hoursNeeded += (pile + k - 1) / k; // Ceiling division } return hoursNeeded <= h; }

Example: Capacity to Ship Packages (LeetCode 1011)

Problem: Ship packages in order within D days. Find minimum ship capacity.
Binary search on: Capacity (not array indices)
public static int shipWithinDays(int[] weights, int days) { int left = 0, right = 0; // Min capacity = max weight, Max capacity = sum of all weights for (int w : weights) { left = Math.max(left, w); right += w; } while (left < right) { int mid = left + (right - left) / 2; if (canShip(weights, mid, days)) { right = mid; // Can ship, try smaller capacity } else { left = mid + 1; // Need larger capacity } } return left; } private static boolean canShip(int[] weights, int capacity, int days) { int daysNeeded = 1, currentWeight = 0; for (int w : weights) { if (currentWeight + w > capacity) { daysNeeded++; currentWeight = w; } else { currentWeight += w; } } return daysNeeded <= days; }

🎯 How to IDENTIFY "Binary Search on Answer" Problems

YOU STRUGGLE HERE! This is the HARDEST pattern to recognize!

🔑 The "Aha!" Moment - Key Recognition Triggers

Pattern Recognition Checklist:

Trigger What to Look For Example
"Minimum ... to satisfy" Find minimum value that works Minimum speed to finish, minimum capacity
"Maximum ... without exceeding" Find maximum value that fits constraint Maximum books to read, maximum weight
"Can you do X in Y time/space?" Optimization problem with constraint Can finish in h hours? Can fit in d days?
"Minimize the maximum" Classic BSOA indicator Minimize max difference, minimize max time
"Maximize the minimum" Another classic indicator Maximize min distance, maximize min value
Given constraint, find optimal Resource limitation problem Given h hours, what's min speed?

💡 The "Monotonic Property" Test

Ask yourself: "If X works, does X+1 also work?"

If YES → Binary Search on Answer!

Example: Koko Eating Bananas
• Can finish in h hours with speed k=5? → NO
• Can finish in h hours with speed k=10? → YES
• Can finish in h hours with speed k=15? → YES (faster always works)
• Can finish in h hours with speed k=20? → YES

Pattern: NO NO NO NO YES YES YES YES
→ We want the FIRST YES! → Binary Search!

🚫 When It's NOT Binary Search on Answer

Red Flags - Don't Use BSOA When:

  1. Array itself is what you're searching → Use regular BS
  2. Need to find EXACT element in sorted array → Template 1
  3. No clear min/max range for answer → Not BSOA
  4. Can't verify a candidate answer efficiently → Not BSOA
  5. Monotonic property doesn't hold → Not BS at all

📝 Step-by-Step Recognition Process

Question Analysis Framework:

STEP 1: Read problem → Extract optimization goal
↳ "Find minimum..."
↳ "Find maximum..."
↳ "Minimize/Maximize..."
STEP 2: Identify what you're optimizing
↳ Is it: speed? capacity? time? distance? threshold?
↳ NOT an element in the array?
STEP 3: Can you define min and max possible values?
↳ YES? → Good sign for BSOA
↳ NO? → Probably not BSOA
STEP 4: Can you write a function: isValid(answer)?
↳ Can I check if answer=X works in O(n)?
↳ YES? → BSOA is feasible
STEP 5: Test monotonic property
↳ If X works, does X+1 work? (for minimum)
↳ If X works, does X-1 work? (for maximum)
↳ YES? → USE BINARY SEARCH ON ANSWER!

6️⃣ Common Pitfalls & How to Avoid Them

⚠️ Pitfall 1: Integer Overflow in mid calculation

// WRONG - Can overflow! int mid = (left + right) / 2; // CORRECT - Safe from overflow int mid = left + (right - left) / 2;

⚠️ Pitfall 2: Infinite Loop in Template 2

// When finding LAST occurrence, need to round UP mid! // WRONG - Can cause infinite loop int mid = left + (right - left) / 2; // Rounds down left = mid; // If left=4, right=5, mid=4 → infinite loop! // CORRECT - Round up when setting left = mid int mid = left + (right - left + 1) / 2; // Rounds up left = mid;

⚠️ Pitfall 3: Wrong Loop Condition

Template Condition Why
Template 1 left <= right Check single element
Template 2 left < right Leave 1 element for post-check
Template 3 left + 1 < right Leave 2 elements for neighbor check

⚠️ Pitfall 4: Forgetting Array is Sorted

Binary search ONLY works on sorted data! If array is not sorted and you can't sort it, you need a different approach.

9️⃣ Time & Space Complexity

Operation Time Complexity Space Complexity Notes
Basic Binary Search O(log n) O(1) Iterative implementation
Recursive Binary Search O(log n) O(log n) Recursion stack space
Find First/Last O(log n) O(1) Same as basic
Rotated Array Search O(log n) O(1) Modified binary search
Binary Search on Answer O(n × log(max-min)) O(1) O(n) to verify each mid
2D Matrix Search O(log(m×n)) O(1) Treat as 1D array
💡 Why log n?
We halve the search space at each step:
n → n/2 → n/4 → n/8 → ... → 1
Number of steps = log₂(n)

🎯 Recognize by: Sorted array, O(log n) needed, find value/position
📊 Three templates: Exact match (<=), Boundaries (<), Neighbors (+1<)
⚡ Complexity: O(log n) time, O(1) space
🔑 Key insight: Eliminate half the search space each iteration
